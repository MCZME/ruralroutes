基于自定义 JSON 多文件加载实现主题模板数据的注册与查询。

## 实现形式

**自定义 JSON 多文件加载**，与 DataMaps 无关。文件和内容完全自控，数据结构贴合业务需求。

**文件位置**：
```
data/<namespace>/ruralroutes/themes/<主题名>.json
```

## 文件格式

```json
{
  "name": "plains_granary",
  "biome": "minecraft:plains",
  "sell_items": [
    { "key": "plains_granary/staples", "items": ["minecraft:bread", "#ruralroutes:candidate/biome/plains/crop"], "pick": 2 },
    "minecraft:oak_planks"
  ],
  "buy_items": [
    { "id": "#ruralroutes:pool/mineral", "pick": 2 },
    "minecraft:iron_ingot"
  ],
  "theme_specialties": [
    "minecraft:golden_carrot"
  ],
  "stock": {
    "default": { "min": 8, "max": 16 },
    "specific": {
      "#ruralroutes:pool/crop": { "min": 32, "max": 64 },
      "minecraft:golden_carrot": { "min": 1, "max": 3 }
    }
  },
  "price_modifiers": {
    "#ruralroutes:candidate/biome/plains/crop": { "sell": 0.82, "buy": 0.76 },
    "#ruralroutes:pool/mineral": { "sell": 1.10, "buy": 1.16 }
  },
  "trade_contracts": [
    {
      "type": "fixed",
      "inputs": [
        { "item": "minecraft:carrot", "count": 8 },
        { "item": "minecraft:bread", "count": 2 },
        { "item": "minecraft:wheat", "count": 6 }
      ],
      "outputs": [
        { "item": "minecraft:golden_carrot", "count": 1 }
      ]
    }
  ]
}
```

### 字段说明

| 字段 | 必需 | 类型 | 说明 |
| :--- | :--- | :--- | :--- |
| `name` | 是 | 字符串 | 主题标识，需与文件名一致 |
| `biome` | 是 | 字符串 | 所属群系 ID |
| `sell_items` | 是 | 数组（字符串或对象） | 村庄出售给玩家的候选物品。支持标签与抽样配置 |
| `buy_items` | 是 | 数组（字符串或对象） | 村庄从玩家收购的候选物品，格式同上 |
| `theme_specialties` | 否 | 字符串数组 | 主题特产，1-2 种物品 ID |
| `stock` | 否 | 对象 | 库存范围配置 |
| `stock.default` | 否 | `{"min", "max"}` | 默认库存范围。不填回退到模组全局配置。 |
| `stock.specific` | 否 | 对象 | 特定物品/标签/候选组来源键的库存范围。键可为 `#tag_id`、`item_id` 或候选组 `key`。 |
| `price_modifiers` | 否 | 对象 | 价格修正系数。键为 `#tag_id` 或 `item_id`。值为 `{"sell", "buy"}`。声明顺序有意义。 |
| `trade_contracts` | 否 | 数组 | 主题提供的交易契约。用于定义动态货币篮交易和固定交换。 |

### 物品引用格式

`sell_items` 和 `buy_items` 支持字符串和对象两种格式：

| 格式 | 示例 | 含义 |
|------|------|------|
| `#<标签ID>` | `#ruralroutes:pool/crop` | 标签引用，运行时展开为所有属于该标签的物品 |
| `<物品ID>` | `minecraft:bread` | 精确物品引用 |
| `{ "id": "#<标签ID>", "pick": N }` | `{ "id": "#ruralroutes:pool/crop", "pick": 3 }` | 单标签候选，展开后随机抽取 `N` 个不同物品 |
| `{ "items": ["<物品ID>", "#<标签ID>", ...], "pick": N }` | `{ "items": ["minecraft:bread", "#ruralroutes:candidate/biome/plains/crop"], "pick": 2 }` | 混合候选组，先合并组内所有物品，再随机抽取 `N` 个不同物品 |
| `{ "key": "<来源键>", "items": [...], "pick": N }` | `{ "key": "plains_granary/staples", "items": ["minecraft:bread", "#ruralroutes:candidate/biome/plains/crop"], "pick": 2 }` | 带稳定来源键的混合候选组，便于 `stock.specific` 按组匹配 |

## 核心规则

**1. # 前缀区分标签与物品**

- `#ruralroutes:pool/crop` → 标签，展开为小麦、胡萝卜、马铃薯等
- `minecraft:bread` → 精确物品

**2. 候选先展开，再按规则抽样**

- 字符串引用：展开后全部纳入候选
- 单引用对象：若声明 `pick`，则从该引用展开结果中随机抽取指定数量
- 混合候选组：先把组内所有标签/物品合并成一个集合，再按 `pick` 抽样
- 抽样在创建节点和周期刷新时执行
- 不同引用命中同一物品时，保留先出现的那条引用

**3. 库存范围随机**

村庄生成时，在 `[min, max]` 内取随机整数作为该物品的库存上限。

匹配优先级：精确 `item` ID > 来源键原样 > 标签去前缀形式 > `default` > 模组全局配置。

**4. 价格修正不校验 buy > sell**

技术上不做约束，允许出现 `buy >= sell`。这是玩法设计的自由。

**5. 主题价格修正只取首个命中规则**

- 精确物品规则优先
- 标签规则按 `price_modifiers` 的声明顺序匹配
- 命中第一条规则后停止，不做主题内叠加

**5.1 价格修正强度建议**

- 本地富产主货：`sell` 常用 `0.78-0.84`，`buy` 常用 `0.72-0.80`
- 次级本地货 / 主题成品：`sell` 常用 `0.88-0.96`，`buy` 常用 `0.82-0.90`
- 输入物资 / 进口需求：`sell` 常用 `1.04-1.10`，`buy` 常用 `1.10-1.18`
- 精确特产：`sell` 常用 `1.08-1.15`，`buy` 常用 `0.84-0.90`

如果同一类物资同时可能出现在 `sell_items` 和 `buy_items` 中，通常应避免 `buy` 高于同类 `sell`，以免形成明显的同站倒卖空间。

**6. 市场事件叠加不在主题模板中处理**

市场波动的多事件叠加由市场模块处理；主题模板只提供单次命中的主题修正。

**7. 物品展开**

`sell_items` / `buy_items` 中的单引用或混合候选组会先展开为候选列表，再根据 `pick` 生成当前周期的实际清单。`stock.specific` / `price_modifiers` 则继续保留对原始标签、物品 ID，或显式候选组 `key` 的匹配能力。由物品池模块通过原版标签 API 提供解析。

## 货币标签

主题模板通过 `withCurrency()` 方法自动添加货币标签到出售池和收购池：

```java
ThemeBuilder.create("plains_granary")
    .biome("minecraft:plains")
    .sell("tag:#ruralroutes:pool/crop")
    .buy("tag:#ruralroutes:pool/mineral")
    .withCurrency()  // 自动添加货币标签到出售池和收购池
    .register();
```

**货币标签定义**：

| 标签 | 包含物品 | 用途 |
|------|----------|------|
| `#ruralroutes:currency` | 铜板、铁币、金币 | 所有货币物品 |
| `#ruralroutes:currency_base` | 铜板 | 基础货币，用于常规交易 |

## 交易契约

主题模板可通过 `trade_contracts` 定义动态货币篮交易和固定交换。

主题模板只声明契约规则，不直接计算价格。动态货币篮契约在运行时调用 `TradePricingService`，因此会受到价值表、主题修正和市场事件影响。未显式声明契约的常规买卖槽位，会继续回退到默认基础货币报价。

### 设计原则

- **固定交换**是“交换”，不是“合成”
- 玩法视角上，固定交换用于主题特产和其他特殊物品的专属交换
- 这些特殊物品不一定珍贵，但必须有明确的主题感
- **动态货币篮**需要专门设计，不应默认把所有币种都塞进同一套规则
- 动态货币篮的关键在于**明确指定** `accepted_currencies` 和与之配套的 `composition`
- 只有当某类物品或某个主题确实需要不同于默认基础货币的报价方式时，才单独声明动态货币篮契约
- 首批动态货币篮只用于**少量专门物品和标签**，不对整个主题做全量覆盖
- 契约是主题的特殊补充，不替代普通 `sell_items` / `buy_items`
- 固定交换的 `outputs` 优先使用主题专属或高辨识度物品，`inputs` 则应是符合主题语义的交换品，而不是强行套用合成配方
- 固定交换应大致接近目标物的基础价值，不要求完全等价，但不宜明显亏本或暴利

### 动态货币篮契约

示例：

```json
{
  "type": "currency_basket_dynamic",
  "side": "sell_to_player",
  "items": [
    "minecraft:iron_pickaxe"
  ],
  "accepted_currencies": [
    "ruralroutes:iron_coin",
    "ruralroutes:copper_coin"
  ],
  "composition": "largest_first"
}
```

字段说明：

| 字段 | 说明 |
|------|------|
| `type` | 固定为 `currency_basket_dynamic` |
| `side` | `sell_to_player` 或 `buy_from_player` |
| `items` | 契约适用物品。当前实现支持精确物品 ID、标签 ID 和通配符 `*` |
| `accepted_currencies` | 货币篮生成器可使用的货币物品，支持物品 ID 和标签 ID |
| `composition` | 货币篮生成策略，如 `smallest_only`、`largest_first`、`single` |

### 固定交换契约

示例：

```json
{
  "type": "fixed",
  "inputs": [
    { "item": "minecraft:diamond", "count": 1 }
  ],
  "outputs": [
    { "item": "ruralroutes:gold_coin", "count": 1 }
  ]
}
```

固定交换契约不调用 `TradePricingService`，因此不受市场事件影响。它只用于特殊配方或明确不随市场变化的交换。

### 当前首批用法

首批主题契约主要采用固定交换，典型包括：

- 粮仓 / 绿洲类的金胡萝卜交换
- 牧场类的鞍
- 工坊类的铁砧
- 采石场类的錾制砂岩
- 染料工坊类的青色染料
- 陶瓦窑类的黄色带釉陶瓦
- 木场 / 船只类的云杉船
- 冰雪 / 狩猎类的蓝冰、营火、骨块

未被契约覆盖的常规买卖槽位，则默认使用基础货币报价，当前默认配置为：

- `accepted_currencies = ["#ruralroutes:currency_base"]`
- `composition = "largest_first"`

首批主题级动态货币篮采用统一的专门设置思路：

- 不为整个主题做全量 `*` 覆盖
- 对矿物采购、铁工具、工坊工具货等少量专门物品或标签，使用 `["ruralroutes:iron_coin", "ruralroutes:copper_coin"] + "largest_first"`
- 其余普通货物继续走默认基础货币回退
- 暂不在首批主题中直接使用金币动态货币篮，避免普通交易过早进入高面额报价

### 当前阶段约束

- 默认常规交易继续保留基础货币回退
- 首批主题模板已经统一补充了专门设计的动态货币篮契约
- 动态货币篮契约只用于确实需要改变币种结构或组合策略的专门场景
- 需要特殊交换感的物品优先使用固定交换
- 交换内容优先从“主题特产 / 主题辨识物”出发，而不是从 vanilla 合成表出发

## 加载与查询

**加载时机**：服务器启动时，扫描所有数据包主题文件夹。

**索引结构**：
- `Map<ResourceLocation, ThemeTemplate>`：主题名 → 模板对象。
- `Map<ResourceLocation, List<ThemeTemplate>>`：群系 ID → 该群系下的主题列表。村庄生成时调用。

**响应 `/reload`**：重新加载所有文件，重建索引。已生成的村庄数据不受影响。

**查询接口**：
- `getTheme(String name)`：按名称获取完整模板。
- `getThemesForBiome(ResourceLocation biome)`：获取指定群系可用主题列表，供村庄生成随机选取。

## 与相关模块的联动

- **物品池**：提供标签展开能力，`sell_items` 等字段依赖物品池动态解析。
- **价值表**：基础价值由价值表模块独立查询，主题模板仅提供修正系数。
- **动态货币篮契约**：主题模板声明货币篮规则，实际价格由交易执行时动态计算。
- **村庄生成**：选中主题后，按模板展开候选、抽样生成当期买卖清单、库存和特产。
- **贸易站 GUI**：读取模板生成的买卖清单和实时库存，显示给玩家。
- **交易引擎**：根据契约生成明确支付计划，再统一执行库存和背包变更。
