# 交易系统架构

## 核心概念

**所有交易本质都是「以物易物」**：物品集合 A → 物品集合 B，即对库存的修改操作。

- 输出物品：从村庄库存扣除
- 输入物品：加入村庄库存

库存本身有 `current` 和 `max` 限制，交易次数自然受库存数量约束。

## 三种契约类型

```mermaid
flowchart LR
    subgraph Types["契约类型"]
        CBD["CurrencyBasketDynamicContract"]
        FT["FixedTradeContract"]
        CX["CoinExchangeContract"]
    end

    subgraph Market["受市场影响"]
        CBD
    end

    subgraph NoMarket["不受市场影响"]
        FT
        CX
    end

    CBD --> TS["贸易站"]
    FT --> DC["展示柜"]
    CX --> CE["货币交换面板"]
```

| 契约 | 是否受市场影响 | 执行载体 | 用途 |
|------|----------------|----------|------|
| `CurrencyBasketDynamicContract` | 是 | 贸易站 | 默认铜板交易、铁币/金币/混合货币动态交易 |
| `FixedTradeContract` | 否 | 展示柜 | 固定特殊配方、剧情化交换 |
| `CoinExchangeContract` | 否 | 贸易站货币交换面板 | 面额兑换（比例动态计算） |

### CurrencyBasketDynamicContract

动态货币篮契约。核心流程：

1. 调用 `TradePricingService` 计算当前市场价格
2. 根据 `acceptedCurrencies` 和 `composition` 策略生成具体货币物品
3. 支持 `smallest_only`（仅铜板）、`largest_first`（混合）、`single`（单一货币）

示例：市场价 137，策略 `largest_first`，货币 `[金币, 铁币, 铜板]` → 报价 `1金币 + 3铁币 + 7铜板`

### FixedTradeContract

固定交换契约。预定义输入输出，不参与价格计算。

执行时直接验证玩家拥有输入物品、村庄拥有输出物品，然后执行转移。

### CoinExchangeContract

货币交换契约。兑换比例通过 `ValueTableManager` 动态计算：

- 向上兑换（低→高）：按价值比计算输入数量，输出固定 1
- 向下兑换（高→低）：输入固定 1，按价值比计算输出数量

## 执行流程

### executeContract（单一契约）

```mermaid
flowchart TD
    A[开始] --> B[获取锁<br/>ChunkPos]
    B --> C{获取成功?}
    C -->|否| D[返回 INVALID_INPUT]
    C -->|是| E[验证玩家输入<br/>validateInput]
    E --> F{验证通过?}
    F -->|否| G[返回 PLAYER_INSUFFICIENT]
    F -->|是| H[契约执行<br/>生成 TradePaymentPlan]
    H --> I[验证村庄库存]
    I --> J{库存充足?}
    J -->|否| K[返回 VILLAGE_INSUFFICIENT]
    J -->|是| L[执行物品转移]
    L --> M[更新村庄库存<br/>updateNodeData]
    M --> N[返回 SUCCESS]
    N --> O[释放锁]
    G --> O
    K --> O
    D --> O
    O --> P[结束]
```

用于展示柜固定交换、货币交换面板兑换。

### executeMixedTrade（混合交易）

```mermaid
flowchart TD
    A[开始] --> B[验证周期一致性<br/>cycleIndex]
    B --> C{周期一致?}
    C -->|否| D[返回 CYCLE_CHANGED]
    C -->|是| E[获取锁<br/>ChunkPos]
    E --> F{获取成功?}
    F -->|否| G[返回 INVALID_REQUEST]
    F -->|是| H[遍历 pendingSlots]
    H --> I[为每个槽位构建支付计划<br/>buildPlanForSlot]
    I --> J[合并所有支付计划<br/>merge + consolidate]
    J --> K[验证玩家库存<br/>validatePlayerInventory]
    K --> L{库存充足?}
    L -->|否| M[返回 PLAYER_INSUFFICIENT]
    L -->|是| N[验证村庄库存<br/>validateVillageInventory]
    N --> O{库存充足?}
    O -->|否| P[返回 VILLAGE_INSUFFICIENT]
    O -->|是| Q[原子执行转移<br/>executeTransfer]
    Q --> R[更新村庄库存]
    R --> S[返回 SUCCESS]
    S --> T[释放锁]
    M --> T
    P --> T
    G --> T
    D --> T
    T --> U[结束]
```

用于贸易站，从 `pendingSlots` 构建支付计划。

## 数据流

### 价格计算链

```mermaid
flowchart TD
    A[ItemStack] --> B["ValueTableManager<br/>queryBaseValue()"]
    B --> C["基础价值<br/>baseValue"]
    C --> D["ThemePriceModifierResolver<br/>resolve()"]
    D --> E["主题修正系数<br/>themeModifier"]
    E --> F["MarketStateResolver<br/>resolvePriceAdjustment()"]
    F --> G["市场波动因子<br/>marketFactor"]
    G --> H["TradePricingService<br/>calculate()"]
    H --> I["最终价格<br/>finalPrice"]
    I --> J["CurrencyBasketComposer<br/>compose()"]
    J --> K["货币物品集合<br/>List&lt;ItemStack&gt;"]
```

### 货币篮生成策略

```mermaid
flowchart TD
    A["finalPrice"] --> B{composition 策略}
    B -->|smallest_only| C["仅使用最低面额<br/>例: 137 → 137铜板"]
    B -->|largest_first| D["优先高面额<br/>例: 137 → 1金币+3铁币+7铜板"]
    B -->|single| E["仅使用指定货币<br/>例: 137铁币 → 14铁币"]
    
    C --> F{包含铜板?}
    D --> F
    E --> F
    
    F -->|是| G["精确表示<br/>无需取整"]
    F -->|否| H["需要取整"]
    H --> I{"side == SELL_TO_PLAYER?"}
    I -->|是| J["向上取整<br/>避免玩家少付"]
    I -->|否| K["向下取整<br/>避免村庄多付"]
```

### 货币判断与价值

- 判断货币：`TagLookupCache.matchesItem(stack, "#ruralroutes:currency")`
- 判断基础货币：`TagLookupCache.matchesItem(stack, "#ruralroutes:currency_base")`
- 获取面额：`ValueTableManager.queryBaseValue(stack)`

货币面额定义：

| 货币 | 面额价值 |
|------|----------|
| `ruralroutes:copper_coin` | 1 |
| `ruralroutes:iron_coin` | 10 |
| `ruralroutes:gold_coin` | 100 |

## 核心数据结构

### TradeContract（接口）

```java
public interface TradeContract {
    TradeContractType type();
    List<Component> getInputDescription();
    List<Component> getOutputDescription();
    boolean validateInput(ServerPlayer player, List<ItemStack> inputs);
    TradeResult execute(ServerLevel level, CommercialNodeData nodeData,
                        ServerPlayer player, List<ItemStack> inputs);
}
```

### TradePaymentPlan（支付计划）

```mermaid
flowchart LR
    subgraph TPP["TradePaymentPlan"]
        PI["playerInputs<br/>玩家支付的物品"]
        PO["playerOutputs<br/>玩家获得的物品"]
        VI["villageInputs<br/>村庄支付的物品"]
        VO["villageOutputs<br/>村庄获得的物品"]
    end
    
    PI --> |"扣除"| Player["玩家背包"]
    PO --> |"加入"| Player
    VI --> |"扣除"| Village["村庄库存"]
    VO --> |"加入"| Village
```

明确的输入/输出物品集合，无抽象数值抵扣。

```java
public record TradePaymentPlan(
    List<ItemStack> playerInputs,    // 玩家支付的物品
    List<ItemStack> playerOutputs,   // 玩家获得的物品
    List<ItemStack> villageInputs,   // 村庄支付的物品
    List<ItemStack> villageOutputs   // 村庄获得的物品
)
```

### TradeResult（统一结果）

```mermaid
flowchart TD
    subgraph Success["成功"]
        S1["outputs: 玩家获得的物品"]
        S2["consumed: 玩家消耗的物品"]
        S3["totalValueExchanged: 交易总价值"]
    end
    
    subgraph Fail["失败"]
        F1["shortfall: 不足物品"]
        F2["reason: 失败原因"]
    end
    
    TR["TradeResult"] --> Success
    TR --> Fail
```

```java
public record TradeResult(
    Reason reason,
    List<ItemStack> outputs,
    List<ItemStack> consumed,
    List<ItemStack> shortfall,
    int totalValueExchanged
)
```

Reason 枚举：SUCCESS, PLAYER_INSUFFICIENT, VILLAGE_INSUFFICIENT, INVALID_INPUT, INVALID_REQUEST, CYCLE_CHANGED

## 规则定义位置

交易契约规则内嵌在**主题模板 JSON** 中。

```json
{
  "trade_contracts": [
    {
      "type": "currency_basket_dynamic",
      "side": "sell_to_player",
      "items": ["#ruralroutes:candidate/theme/plains_workshop/tool_goods"],
      "accepted_currencies": [
        "ruralroutes:iron_coin",
        "ruralroutes:copper_coin"
      ],
      "composition": "largest_first"
    },

      "type": "fixed",
      "inputs": [{ "item": "minecraft:diamond", "count": 1 }],
      "outputs": [{ "item": "minecraft:emerald", "count": 3 }]
    }
  ]
}
```

## 并发控制

```mermaid
flowchart LR
    subgraph TLM["TradeLockManager"]
        Locks["1024 个锁槽位<br/>ReentrantLock[]"]
    end
    
    Request["交易请求"] --> Chunk["ChunkPos"]
    Chunk --> Hash["hash = x ^ z"]
    Hash --> Index["index = hash % 1024"]
    Index --> Lock["locks[index]"]
    
    Lock -->|"tryLock(5s)"| Acquire["获取锁"]
    Acquire -->|"成功"| Execute["执行交易"]
    Acquire -->|"失败/超时"| Fail["返回失败"]
    Execute -->|"完成"| Release["释放锁"]
```

- 同一区块的交易请求串行执行
- 不同区块可并行
- 1024 个锁槽位
- 5 秒超时
