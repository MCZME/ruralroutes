## 概述

市场状态系统负责在每个贸易周期生成并维护当前周期唯一的 `MarketState`。

当前实现采用**规则驱动市场事件池**：

- 数据包提供 `MarketEventRule`
- 服务端在周期刷新时按权重抽取规则
- 规则按作用域展开为具体 `MarketEvent`
- 所有事件汇总为当前周期的 `MarketState`
- 定价、传闻板等系统都只读取这份状态

## 核心类

### 数据模型（`core/market/`）

| 类 | 说明 |
|---|-----|
| `MarketScopeType` | 作用域枚举：`GLOBAL` / `BIOME` / `THEME` |
| `MarketEvent` | 当前周期实际生效的市场事件 record，包含传闻元数据 |
| `MarketState` | 当前周期市场状态快照 record |
| `MarketContext` | 查询时使用的主题 / 群系上下文 |
| `MarketPriceAdjustment` | 市场解析后的价格调整结果 |
| `MarketEventScopeRule` | 单条规则的作用域展开描述 |
| `MarketEventRule` | 市场事件规则模板，支持传闻语义字段 |

### 核心组件

| 类 | 说明 |
|---|-----|
| `MarketEventRuleCatalog` | 规则目录，负责加载数据包 JSON |
| `MarketStateGenerator` | 规则抽取、作用域展开与状态生成 |
| `MarketStateResolver` | 解析指定物品在指定上下文下的价格调整 |

## 数据流

```text
market_event_rules/*.json
        ↓
MarketEventRuleCatalog
        ↓
按权重抽取规则
        ↓
MarketStateGenerator
        ↓
展开为 MarketEvent 列表
        ↓
生成 MarketState
        ↓
CycleManager 持有与持久化
        ↓
TradePricingService / RumorBoard / 调试命令读取
```

## 规则模型

规则文件路径：

`data/<namespace>/ruralroutes/market_event_rules/*.json`

基础格式：

```json
{
  "id": "ruralroutes:common/food_shortage",
  "name_key": "market.ruralroutes.common.food_shortage",
  "target_ref": "#ruralroutes:pool/food",
  "rumor_family": "shortage",
  "rumor_target_key": "rumor.target.food",
  "scopes": [
    { "type": "global", "selector": "all" }
  ],
  "delta": 0.15,
  "weight": 100
}
```

字段说明：

| 字段 | 说明 |
|---|-----|
| `id` | 规则唯一 ID |
| `name_key` | 本地化 key |
| `target_ref` | 物品 ID 或标签引用 |
| `scopes` | 一条或多条作用域展开规则 |
| `delta` | 涨跌幅，正涨负跌 |
| `weight` | 抽取权重，缺省时使用默认值 |
| `rumor_family` | 可选，传闻语义族 |
| `rumor_target_key` | 可选，传闻使用的玩家视角目标名称 |

## 作用域展开

`MarketEventScopeRule` 支持两类选择器：

| 选择器 | 说明 |
|---|-----|
| `ALL` | 展开到该作用域下所有可用目标 |
| `LIST` | 仅展开到显式给定目标列表 |

展开规则：

- `GLOBAL`：生成一个全局事件
- `BIOME + ALL`：展开到当前已加载主题所覆盖的全部群系
- `BIOME + LIST`：展开到显式列出的群系
- `THEME + ALL`：展开到当前全部主题
- `THEME + LIST`：展开到显式列出的主题

因此：

- “抽中了几条规则”不等于“最终有几个事件”
- 一条规则可能展开为多个事件

## 生成流程

`MarketStateGenerator` 在生成状态时执行：

1. 检查 `market.enabled`
2. 检查规则目录是否已加载内容
3. 获取当前可用群系与主题列表
4. 按 `market.rule_pick_count` 进行不放回加权抽取
5. 将选中的规则按作用域展开为 `MarketEvent`
6. 将 `rumor_family` / `rumor_target_key` 复制到运行时事件
7. 按事件键去重
8. 生成 `MarketState(cycleIndex, events)`

## 确定性

`generateDeterministic(cycleIndex)` 使用周期索引构造随机种子。

目的：

- 同一周期内状态稳定
- 服务端重启后可重建相同周期状态
- 传闻板与定价读取同一份权威结果

前提是当前规则集、主题集和展开逻辑保持不变。

## 周期接入

市场状态通过 `CycleManager` 与贸易周期系统接入。

### `CycleManager`

负责：

- 缓存当前周期索引
- 持久化 `MarketState`
- 在周期变化时生成新状态
- 在节点刷新或定价时兜底初始化状态

### 更新时机

1. **周期边界**
   - `RuralRoutes.onServerTick()` 调用 `updateCurrentCycle()`
   - 若周期推进，则调用 `getOrInitMarketState()`

2. **节点刷新兜底**
   - `CommercialNodeManager.checkAndRefreshCycle()` 在节点刷新前调用 `getOrInitMarketState()`
   - 确保当前周期一定存在市场状态

3. **定价读取**
   - `TradePricingService` 查询价格时，从 `CycleManager` 读取当前 `MarketState`

## 查询流程

`MarketStateResolver.resolvePriceAdjustment()` 的处理顺序：

1. 遍历当前周期全部事件
2. 用 `TagLookupCache.matchesItem()` 检查 `target_ref` 是否命中当前物品
3. 用 `MarketContext` 检查作用域是否命中当前节点
4. 收集全部命中事件
5. 将所有 `delta` 加法叠加
6. 应用 `market.max_delta` 上限
7. 返回 `MarketPriceAdjustment`

## 配置项

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `market.enabled` | 是否启用市场系统 | true |
| `market.rule_pick_count` | 每周期抽取规则数量 | 3 |
| `market.max_delta` | 最大涨跌幅度 | 0.50 |

## 管理命令

| 命令 | 说明 |
|------|------|
| `/ruralroutes refresh` | 强制刷新贸易周期 |
| `/ruralroutes market refresh` | 强制刷新市场状态 |
| `/ruralroutes market info` | 显示当前市场信息 |

## 当前边界

当前实现已确定的边界：

- 规则模板由数据包驱动
- 运行时状态由 `CycleManager` 持有
- 价格系统只读取 `MarketState`，不自行生成事件
- 传闻板只消费服务端下发的 `MarketState`
- 传闻层依赖 `rumor_family` / `rumor_target_key` 生成玩家文本

当前未在本模块处理的内容：

- 单条规则内容平衡
- 事件文本美术与本地化扩写
- 恢复规则 `category` / `incremental`
