# 区块数据存储实现

## 数据结构

每个区块存储一个 `CommercialNodeData`，当前结构已经从“裸物品 ID + 简单库存数字”升级为“展示索引 + 组件感知库存键”双层表达。

| 字段 | 类型 | 说明 |
|------|------|------|
| `tradeNodeId` | UUID | 节点唯一标识 |
| `themeName` | ResourceLocation | 主题标识 |
| `sellItems` | List<NodeTradeEntry> | 当前周期出售清单 |
| `buyItems` | List<NodeTradeEntry> | 当前周期收购清单 |
| `stocks` | Map<TradeItemKey, NodeStockEntry> | 共享库存池 |
| `refreshTimestamp` | long | 上次刷新时间戳 |

## 两层表达

### NodeTradeEntry

用于有序展示视图：

| 字段 | 类型 | 说明 |
|------|------|------|
| `sourceKey` | String | 候选来源键，用于主题修正、市场规则、契约匹配 |
| `stockKey` | TradeItemKey | 指向库存条目的稳定地址 |

`NodeTradeEntry` 负责：

- 保留 UI 展示顺序
- 提供当前物品的展示栈
- 让同一基础物品的不同组件变体能在列表中并存

### NodeStockEntry

用于库存条目：

| 字段 | 类型 | 说明 |
|------|------|------|
| `stack` | ItemStack | 规范化物品栈原型，数量固定为 1 |
| `current` | int | 当前净持有量 |
| `max` | int | 总容量上限 |

在当前实现中：

- `current` 对应出售侧现货量
- `max` 对应共享库存池总容量，等于出售现货量 + 收购容量
- 普通物品通常保持 `0 <= current <= max`
- 货币物品在交易入库时允许 `current > max`，用来保留超过本期收购容量的真实收款结果，避免货币被虚空吞掉

### TradeItemKey

库存索引键，承载：

- `itemId`
- 可选的组件签名 `componentSignature`

用途：

- 作为 `stocks` map 的稳定键
- 作为市场匹配、主题价格修正、混合交易合并与扣除的统一身份

## 读取规则

`CommercialNodeData.getStock(...)` 提供一层兼容读取：

- 先尝试精确 `TradeItemKey`
- 若请求键带组件且未命中，再回退到同 `itemId` 的无组件键

这个回退只用于“库存读取容错”，不会改变库存内部的真实索引结构。

## 存储方案

使用 NeoForge 的 Attachment 系统存储区块数据：

- 注册 `AttachmentType<CommercialNodeData>`
- 数据随区块自动持久化
- 通过 `chunk.getData()` / `chunk.setData()` 访问

## 数据存储位置对比

| 数据 | 存储位置 | 用途 |
|------|----------|------|
| `tradeNodeId` | BlockEntity + 区块数据 | 唯一标识，双重存储保证一致性 |
| `themeName` | BlockEntity + 区块数据 | BlockEntity 用于校验，区块数据用于运行 |
| `sellItems` | 区块数据 | 当前周期出售清单与展示栈 |
| `buyItems` | 区块数据 | 当前周期收购清单与展示栈 |
| `stocks` | 区块数据 | 核心库存数据 |
| `refreshTimestamp` | 区块数据 | 周期刷新判断 |

## BlockEntity 角色

贸易站 BlockEntity 保留校验数据：

- `villageTheme`
- `tradeNodeId`

展示柜和传闻板 BlockEntity 仅保留节点定位信息：

- `tradeNodeId`
- `tradeStationPos`

展示柜实际展示的 `ItemStack` 仍存储在自己的 BlockEntity 中，但来源于 `CommercialNodeData.sellItems`。

## 创建流程

玩家首次交互时触发：

```text
玩家右键贸易站
    ↓
检查 BlockEntity.themeName 是否有效
    ↓
检查区块是否已有 CommercialNodeData
    ↓
无数据时创建：
  - tradeNodeId = UUID.randomUUID()
  - themeName = BlockEntity.themeName
  - 读取 ResolvedTheme
  - 对 sellItems / buyItems 执行候选展开与抽样
  - 为每个入选项生成 NodeTradeEntry
  - 为每个库存项生成 NodeStockEntry
  - refreshTimestamp = 当前时间
    ↓
同步写入贸易站、展示柜、传闻板的 tradeNodeId
```

## 库存初始化规则

库存只为“本期实际入选的物品”创建条目：

- 出售物品：生成初始现货量
- 仅收购物品：生成收购容量但 `current = 0`
- 同一物品同时出现在买卖两侧：共享同一个 `TradeItemKey` 库存池

库存上限来源：

1. `ResolvedTheme.stock`
2. 当前周期市场库存修正
3. 节点级共享库存合并规则

## 周期刷新

节点在交互触发的刷新检查中按需惰性刷新：

1. 检查 `refreshTimestamp` 是否落后于当前周期
2. 重新从 `ResolvedTheme` 抽样本期买卖清单
3. 重新初始化 `stocks`
4. 更新 `refreshTimestamp`

刷新后：

- 上一周期的实时库存不会保留
- 新周期会重新应用当前 `MarketState` 的库存修正
- 组件化物品的展示栈与库存键会重新按本期清单生成

## 规范化规则

`CommercialNodeData` 在构造时会规范化 `stocks`：

- map key 最终以 `NodeStockEntry.stockKey()` 为准
- `NodeStockEntry.stack` 会被强制拷贝并归一化为 `count = 1`

这样可以避免：

- 外部传入键和栈内容不一致
- 同一物品因数量不同产生不同库存身份

## 与运行时模块的关系

- **贸易站 GUI**：读取 `NodeTradeEntry.displayStackOrDefault()` 进行展示
- **市场定价**：读取 `themeName` 和展示栈，再配合 `sourceKey` 解析规则
- **展示柜**：从 `sellItems` 中随机选取展示物
- **交易执行**：直接修改 `stocks` 中的 `NodeStockEntry`
- **货币入库例外**：货币条目在增加库存时允许超过 `max`，但周期刷新后仍会按当期基线重新初始化
