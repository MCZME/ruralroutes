# 传闻板实现

## 数据结构

### BlockEntity

`RumorBoardBlockEntity` 存储节点ID和布局数据：

| 字段 | 类型 | 说明 |
|---|---|---|
| `tradeNodeId` | `UUID` | 节点唯一标识，由贸易站同步 |
| `layoutCycleIndex` | `long` | 布局对应的周期索引 |
| `layouts` | `List<StickyNoteLayout>` | 便签布局列表 |

情报数据通过确定算法从全局市场状态生成，无需存储。

**注意**：传闻板不需要 `villageTheme`，仅通过 `tradeNodeId` 与贸易站关联。

### 便签布局数据结构

`StickyNoteLayout` 记录：

| 字段 | 类型 | 说明 |
|---|---|---|
| `index` | `int` | 便签索引 |
| `offsetX` | `float` | X 偏移（0.0-1.0，相对于 GUI 宽度）|
| `offsetY` | `float` | Y 偏移（0.0-1.0，相对于 GUI 高度）|
| `rotation` | `float` | 旋转角度（度）|
| `layer` | `int` | 层级：0=底层, 1=顶层 |

### 数据同步机制

BlockEntity 实现标准数据同步：

- `TradeNodeBlockEntity.getUpdateTag()` - 同步 `tradeNodeId` 和 `tradeStationPos` 到客户端
- `RumorBoardBlockEntity.getUpdateTag()` - 额外同步 `layoutCycleIndex` 和 `rumorCycleIndex`
- `handleUpdateTag()` - 客户端接收更新
- `setTradeNodeInfo()` / `setTradeNodeId()` - 修改后调用 `sendBlockUpdated()` 触发同步

`MarketState` 不存储在传闻板 `BlockEntity` 中，而是在玩家打开传闻板时由服务端通过 `OpenRumorBoardPayload` 发送给客户端。便签布局和传闻缓存仍由客户端按需生成并缓存到本地 `BlockEntity` 实例中。

## 初始化流程

传闻板不需要预置主题，仅通过贸易站自动同步节点ID：

### 阶段一：结构模板（开发时）

执行者：模组开发者
时机：制作 .nbt 结构模板时
操作：放置传闻板方块即可，无需配置
确定数据：无

### 阶段二：首次交互（运行时）

执行者：贸易站（自动同步）
时机：玩家首次激活贸易站时
操作：贸易站创建区块数据后同步 `tradeNodeId` 到附近传闻板
确定数据：`tradeNodeId`

## 交互流程

```
玩家右键传闻板
    │
    ├─ 客户端：
    │   检查 `tradeStationPos`
    │   ├─ 未同步 → 不发起有效交互
    │   └─ 已同步 → 等待服务端校验结果
    │
    └─ 服务端：
        检查区块是否已有 CommercialNodeData
        ├─ 无 → 提示"请先激活贸易站"
        ├─ `tradeNodeId` 不匹配 → 提示"数据不匹配"
        └─ 通过 → 发送 `OpenRumorBoardPayload(pos, marketState)`
```

当前实现优先简化只读界面的打开链路，因此传闻板不再注册独立 `MenuType`，也不再通过 `player.openMenu()` 打开 GUI；但市场状态本身仍以服务端为准。

## GUI 实现

### 设计约束

- 数据来源：服务端当前周期的 `MarketState`
- 空状态处理：显示闲谈文本
- 排序规则：原始顺序，不额外排序
- 交互：只读，无交易操作
- 数据获取：**服务端同步市场状态，客户端生成展示文本与布局**
- 打开方式：**服务端校验后发送 payload，客户端打开纯 `Screen`**

### 当前实现说明

- `RumorBoardScreen` 自行收集展示数据并渲染，不依赖 `RumorBoardMenu`
- `MarketState` 由服务端在打开时通过 `OpenRumorBoardPayload` 下发
- 周期索引与剩余时间统一通过 `CycleManager.getEffectiveTime(level)` 计算
- 客户端只基于服务端下发的 `MarketState` 生成情报文本，不再自行重算市场状态

### 已确定事项

| 项目 | 决定 |
|------|------|
| 周期时间 | 显示模糊描述（如"约 1 天后"），无需实时更新 |
| 布局风格 | 便签风格，杂乱分布 |
| 数据范围 | 全局市场状态（所有传闻板内容相同） |
| 卡片内容 | 情报完整文本 |
| 空状态 | 闲谈文本 |
| 卡片风格 | 统一风格，不区分涨跌颜色 |
| 便签尺寸 | 固定大小（80×60）|
| 分层逻辑 | 奇偶交替（第 1、3、5 张底层，第 2、4、6 张顶层）|
| 布局存储 | 存储在 `RumorBoardBlockEntity` 中（每个方块独立布局）|
| 更新时机 | 周期刷新时生成布局，同一周期内不变 |

### GUI 尺寸

- 固定尺寸：280×200
- 背景风格：深棕木纹（`0xCC4A3728`）

### 便签视觉细节

- 便签背景颜色：多种柔和颜色循环使用（浅黄、浅橙、浅红、浅粉、浅紫、浅蓝、天蓝、浅绿）
- 边框样式：轻微深色描边
- 字体颜色：深灰（`0x333333`）
- 图钉效果：红色图钉在顶部中心

### 层内布局算法

```java
// 布局参数
MIN_OFFSET_X = 0.05f    // 最小 X 偏移（避免贴边）
MAX_OFFSET_X = 0.85f    // 最大 X 偏移
MIN_OFFSET_Y = 0.08f    // 最小 Y 偏移（留出标题空间）
MAX_OFFSET_Y = 0.82f    // 最大 Y 偏移
ROTATION_RANGE = 15.0f  // 最大旋转角度（±15°）
COLLISION_THRESHOLD_X = 0.15f  // X 方向最小间距
COLLISION_THRESHOLD_Y = 0.18f  // Y 方向最小间距
MAX_ATTEMPTS = 100      // 最大重试次数
```

## 类设计

### 核心类

| 类 | 职责 |
|------|------|
| `StickyNoteLayout` | 便签布局数据结构，支持 NBT 序列化 |
| `StickyNoteLayoutGenerator` | 布局生成算法，确定性种子 |
| `OpenRumorBoardPayload` | 服务端向客户端发送当前周期 `MarketState` 并触发打开界面 |
| `RumorBoardBlock` | 服务端校验激活状态与节点一致性，发送 `OpenRumorBoardPayload` |
| `RumorBoardScreen` | 基于服务端下发的 `MarketState` 收集展示数据并渲染便签 |
| `StickyNoteWidget` | 单个便签渲染，支持旋转 |

### 数据流

```
玩家右键传闻板
    │
    └─ 服务端校验节点数据
        │
        ├─ 未激活 / 不匹配 → 提示消息
        │
        └─ 校验通过 → 发送 `OpenRumorBoardPayload`
                        │
                        ▼
                    客户端打开 `RumorBoardScreen`
                        │
                        ▼
                    RumorBoardScreen.collectClientData()
                        │
                        ├─ 计算周期索引（从 `marketState.cycleIndex()`）
                        ├─ 基于服务端下发的 `MarketState` 生成情报列表（`RumorGenerator`）
                        ├─ 生成/获取布局（`StickyNoteLayoutGenerator`）
                        └─ 缓存到客户端 `RumorBoardBlockEntity`
```
