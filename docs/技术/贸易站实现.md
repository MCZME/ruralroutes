# 贸易站实现

## BlockEntity 与区块数据分工

`TradeStationBlockEntity` 只存储校验字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `villageTheme` | `ResourceLocation` | 主题标识 |
| `tradeNodeId` | `UUID` | 节点唯一标识 |

核心业务数据存放在区块的 `CommercialNodeData` 中。

## 运行时数据模型

### CommercialNodeData

贸易站当前不再直接读取“裸物品 ID 列表”，而是消费：

- `sellItems: List<NodeTradeEntry>`
- `buyItems: List<NodeTradeEntry>`
- `specialties: List<NodeTradeEntry>`
- `stocks: Map<TradeItemKey, NodeStockEntry>`

### NodeTradeEntry

每个交易项同时携带：

- `sourceKey`
- `stockKey`

这样 GUI 可以拿到：

- 当前物品的展示栈
- 该物品在主题候选中的来源键
- 它对应的库存键

## TradeSlot

`TradeSlot` 仍是不可交互展示槽位，但它承载的数据已经和过去不同：

| 字段 | 类型 | 说明 |
|------|------|------|
| `itemId` | `ResourceLocation` | 基础物品 ID |
| `displayStack` | `ItemStack` | 当前展示栈，可带组件 |
| `baseStock` | `int` | 出售区为当前库存，收购区为已收购数量 |
| `maxStock` | `int` | 收购区最大容量 |
| `pendingCount` | `int` | 暂存数量 |
| `price` | `int` | 当前基础价格 |
| `priceStacks` | `List<ItemStack>` | 货币篮报价 |
| `inputStacks` | `List<ItemStack>` | 固定交换输入 |
| `tradeType` | `TradeContractType` | 契约类型 |
| `isBuy` | `boolean` | true=玩家买入，false=玩家卖出 |

### 重要变化

- `displayStack` 可能来自带组件的 `NodeTradeEntry`
- 价格计算时会带上 `sourceKey`
- 库存读取时按 `entryData.tradeItemKey()` 访问，而不是按裸 `itemId`

## 槽位初始化

服务端 `initializeFromNodeData()` 会：

1. 读取 `CommercialNodeData.sellItems()` / `buyItems()`
2. 读取 `stocks`
3. 通过 `ThemeManager.getTheme(themeName)` 获取 `ResolvedTheme`
4. 为每个 `NodeTradeEntry` 创建 `TradeSlot`

初始化时对每个槽位执行：

- `slot.setDisplayStack(entryData.displayStackOrDefault())`
- `slot.setBaseStock(stocks.get(entryData.tradeItemKey()).current())`
- 调用 `calculateSellPrice(...)` / `calculateBuyPrice(...)`
- 把 `entryData.sourceKey()` 一并传入定价服务

## 契约匹配

`findMatchingContract()` 现在基于：

- `ResolvedTheme.tradeContracts()`
- `NodeTradeEntry`
- 当前展示栈

动态货币篮匹配规则支持：

- `*`
- 标签
- 精确物品
- `@source_key`

其中 `@source_key` 由 `matchesContractPattern()` 处理。

## 默认货币篮价格

未命中显式契约时：

- 使用当前展示栈
- 带上 `sourceKey`
- 调用 `TradePricingService.calculateFinalPrice(...)`
- 再用默认基础货币配置生成 `priceStacks`

这样默认报价也能区分：

- 同一基础物品的不同组件变体
- 同一基础物品来自不同候选组的不同语义来源

## 暂存区

`pendingSlots` 仍然是服务端权威的暂存区。

关键点：

- 新增条目时复制来源槽位的 `displayStack`
- 因此组件化物品在暂存区内会保持原始身份
- 后续混合交易执行器会按完整 `TradeItemKey` 合并和校验

## 确认交易流程

```text
客户端发送 TradeRequestPayload(CONFIRM)
    ↓
服务端 TradeStationMenu.executeTrade()
    ↓
读取当前 CommercialNodeData
    ↓
TradeContractExecutor.executeMixedTrade(
    level,
    nodeData,
    player,
    pendingSlots,
    currentCycleIndex,
    blockPos
)
    ↓
成功：
  - 清空 pendingSlots
  - 读取最新 nodeData
  - 更新槽位库存
  - 同步槽位、暂存区、货币库存

失败：
  - 周期变化时重建整个清单
  - 其他失败时仅发送 GUI 反馈
```

### 执行器侧的新约束

混合交易执行器现在会：

- 按 `TradeItemKey` 合并支付计划中的物品
- 优先消耗带组件需求
- 避免组件物品与普通同类物品重复计数

这保证了贸易站 GUI 上展示的带组件商品，结算时不会被错误地替换成任意同类物品。

## 货币交换

货币交换仍然独立于 `pendingSlots`：

```text
CoinExchangeWidget
    → CoinExchangeRequestPayload
    → TradeStationMenu.executeCoinExchange()
    → TradeContractExecutor.executeContract()
```

村庄货币库存读取方式也已经切换到节点库存层：

- 通过 `TradeItemKey.of(itemId)` 读取 `stocks`
- 返回对应 `NodeStockEntry.current()`

## 首次交互创建节点

```text
玩家右键贸易站
    ↓
检查 BlockEntity.themeName
    ↓
若区块无 CommercialNodeData：
  - 调用 CommercialNodeManager.createNodeData()
  - 内部读取 ResolvedTheme
  - 生成 sellItems / buyItems / specialties / stocks
  - 同步 tradeNodeId 到附近展示柜和传闻板
    ↓
打开 GUI
```

## 同步要点

| 场景 | 同步内容 |
|------|----------|
| 打开 GUI | 全量槽位 + 村庄货币库存 |
| 增删暂存项 | 暂存区增量同步 |
| 常规交易成功 | 全量槽位 + 暂存区 + 货币库存 |
| 货币交换 | 货币库存 + GUI 反馈 |
| 周期刷新 | 重建槽位后全量同步 |
