# 贸易站实现

## 数据结构

### BlockEntity

`TradeStationBlockEntity` 仅存储校验数据，不存储核心业务数据：

| 字段 | 类型 | 说明 |
|---|---|---|
| `villageTheme` | `ResourceLocation` | 主题标识，用于校验 |
| `tradeNodeId` | `UUID` | 节点唯一标识 |

核心业务数据（库存、刷新时间戳）存储在区块的 `CommercialNodeData` 中。

### 数据同步机制

BlockEntity 实现标准数据同步：

- `getUpdateTag()` - 同步 `villageTheme` 到客户端
- `handleUpdateTag()` - 客户端接收更新
- `setVillageTheme()` - 修改后调用 `sendBlockUpdated()` 触发同步

### TradeSlot 设计

`TradeSlot` 是不可交互的展示槽位：

| 字段 | 类型 | 说明 |
|------|------|------|
| `itemId` | `ResourceLocation` | 物品ID |
| `displayStack` | `ItemStack` | 物品类型模板 |
| `baseStock` | `int` | 原始库存（出售区）或已收购数量（收购区） |
| `maxStock` | `int` | 最大库存（收购区使用） |
| `pendingCount` | `int` | 暂存数量（用于实时扣减显示） |
| `price` | `int` | 基础价格 |
| `priceStacks` | `List<ItemStack>` | 货币篮报价（动态货币篮契约） |
| `inputStacks` | `List<ItemStack>` | 输入物品（固定交换契约） |
| `tradeType` | `TradeContractType` | 普通交易类型（CURRENCY_BASKET_DYNAMIC/FIXED） |
| `isBuy` | `boolean` | true=购买（村庄卖给玩家），false=出售（玩家卖给村庄） |

计算属性：
- 出售区：`stockCount = baseStock - pendingCount`
- 收购区：`stockCount = maxStock - baseStock - pendingCount`

特点：
- `mayPickup()` 返回 false — 不可拿起
- `mayPlace()` 返回 false — 不可放入
- 支持叠加逻辑和实时库存扣减显示

### PendingTradeSlot

暂存区专用槽位，继承 `TradeSlot`：

| 额外字段 | 说明 |
|----------|------|
| `sourceSlotIndex` | 来源槽位索引（-1 表示无） |
| `sourceIsBuy` | 来源是否为出售区 |

用于追踪交易来源，取消时恢复来源槽位的 `pendingCount`。

### 槽位管理

Menu 中槽位独立管理：

```java
List<TradeSlot> sellSlots;           // 出售槽位（村庄卖给玩家）
List<TradeSlot> buySlots;            // 收购槽位（玩家卖给村庄）
List<PendingTradeSlot> pendingSlots; // 暂存区槽位（多物品）
```

## 交易报价卡片

### TradeOfferCardWidget

交易报价卡片采用两行紧凑布局：

```text
第一行：物品图标(14×14) + 库存数量
第二行：货币篮/输入物品 + 价格/个
```

**尺寸常量**：
- `CARD_WIDTH = 65`
- `CARD_HEIGHT = 26`
- `ITEM_ICON_SIZE = 14`
- `CURRENCY_ICON_SIZE = 10`

**状态展示**：
- 正常：默认半透明背景
- 悬停：背景高亮
- 已加入暂存：选中边框
- 库存不足：数量显示红色

**货币篮展示规则**：
- 最多显示 2 种货币图标
- 超出部分显示 `+N`

**Tooltip 分区**：
- 物品图标区域：显示原版物品 tooltip（附魔、描述等）
- 其他区域：显示自定义 tooltip（库存、价格明细）

## 交易暂存区

### TradeAreaWidget

暂存区采用两栏布局：

```text
┌─────────────────────────────────────────┐
│ 玩家想要的          玩家支付的           │
│ ┌────┐┌────┐       ┌────┐┌────┐        │
│ │物品││货币│       │物品││货币│        │
│ └────┘└────┘       └────┘└────┘        │
│              [确认]                      │
└─────────────────────────────────────────┘
```

**左侧"玩家想要的"**：
- 买入的物品
- 卖出获得的货币（货币篮类型）

**右侧"玩家支付的"**：
- 卖出的物品
- 买入支付的货币（货币篮类型）或输入物品（固定交换类型）

**卡片布局**：
- 双行横向滚动
- 内容超出后通过滚动条浏览，不再限制为固定 4 张卡片

**点击交互**：
- 左右两侧的物品卡片都可点击取消
- 点击后触发 `onRemoveSlot` 回调

### 设计理念

暂存区 = 购物车/订单，一笔交易可以包含多个物品：

- 一笔交易可以**同时包含买入和卖出**物品
- 每个物品独立记录交易类型（买入/卖出）
- 结算时由 `TradeContractExecutor` 统一处理

## 货币交换面板

### 面板结构

`CoinExchangeWidget` 现已从占位按钮升级为独立的右侧面板：

- 顶部：玩家钱包条，常驻显示铜板、铁币、金币持有量
- 底部：3 个货币筹码，分别对应铜板、铁币、金币
- 交互：点击某个货币筹码后，在其上方弹出“使用该货币可执行的兑换”选项
- Tooltip：显示玩家持有量、村庄可供量、最大可兑换次数

### 交互规则

- 单击：兑换 1 次
- `Shift + 单击`：在玩家库存和村庄库存允许的范围内尽可能全部兑换
- 玩家不足和村庄不足使用不同禁用态
- 结果反馈复用 `TradeFeedbackPayload`，直接显示在贸易站 GUI 内

### 服务端执行链路

货币交换不进入 `pendingSlots`，而是单独走请求链路：

```text
CoinExchangeWidget
    -> CoinExchangeRequestPayload
    -> TradeStationMenu.executeCoinExchange()
    -> TradeContractExecutor.executeContract()
    -> CoinExchangeStatePayload + TradeFeedbackPayload
```

其中：

- 玩家钱包数量由客户端本地背包直接统计
- 村庄可供数量由服务端读取 `CommercialNodeData.stocks`
- 每次兑换完成后都会重新同步村庄货币库存

### 交易执行

确认时服务端直接从 `pendingSlots` 读取交易数据：

```
客户端发送 TradeRequestPayload(CONFIRM)
    ↓
服务端 executeTrade()
    ↓
直接从 this.pendingSlots 读取
    ↓
TradeContractExecutor.executeMixedTrade(level, nodeData, player, pendingSlots, cycleIndex, blockPos)
    ↓
服务端重新计算每个物品的权威价格
    ↓
验证周期一致性、库存
    ↓
原子执行转移
```

**关键设计**：服务端已有 `pendingSlots` 数据，不需要额外发送订单结构。

### 交互逻辑

| 操作 | 行为 |
|------|------|
| 点击出售区卡片 | 发送 ADD_BUY 请求，添加到暂存区 |
| 点击收购区卡片 | 发送 ADD_SELL 请求，添加到暂存区 |
| 点击左侧卡片 | 发送 REMOVE_BUY 请求，移除暂存项 |
| 点击右侧卡片 | 发送 REMOVE_SELL 请求，移除暂存项 |
| 点击确认 | 发送 CONFIRM 请求，执行交易 |

## 初始化流程

贸易站方块采用**两阶段初始化**：

### 阶段一：结构模板（开发时）

执行者：模组开发者
时机：制作 .nbt 结构模板时
操作：使用配置工具预置 `themeName`
确定数据：`themeName`

### 阶段二：首次交互（运行时）

执行者：玩家
时机：右键核心方块时
操作：检查并创建区块数据
确定数据：当期买卖清单与库存数据

## 首次交互流程

```
玩家右键贸易站
    │
    ▼
检查 BlockEntity.themeName 是否有效
    │
    ├─ 无效 → 无响应
    │
    ▼ 有效
检查区块是否已有 CommercialNodeData
    │
    ├─ 有 → 校验 tradeNodeId，打开 GUI
    │
    ▼ 无
检查旁边是否有村庄
    │
    ├─ 无 → 无响应
    │
    ▼ 有
创建 CommercialNodeData：
  - tradeNodeId = UUID.randomUUID()
  - themeName = BlockEntity.themeName
  - sellItems = 从 ThemeTemplate.sellItems 展开候选并抽样
  - buyItems = 从 ThemeTemplate.buyItems 展开候选并抽样
  - stocks = 仅为本次入选物品初始化（支持叠加）
  - refreshTimestamp = 当前时间
同步写入 BlockEntity.tradeNodeId（贸易站、展示柜、传闻板）
打开 GUI
```

## 节点同步机制

首次交互时，贸易站会将 `tradeNodeId` 同步到附近的展示柜和传闻板：

- 同步范围：贸易站周围 16 格半径
- 同步对象：`DisplayCaseBlockEntity`、`RumorBoardBlockEntity`
- 同步时机：创建 `CommercialNodeData` 后立即同步

这确保同一建筑内的三个方块共享同一个商业节点。展示柜和传闻板不需要 `villageTheme`，仅通过 `tradeNodeId` 与贸易站关联。

## 数据同步流程

### 打开 GUI 时同步

```
服务端 openMenu()
    │
    ▼
createMenu() 创建 TradeStationMenu
    │
    ▼
Menu 构造时从 CommercialNodeData 初始化槽位
    │
    ▼
openMenu() 返回后发送 TradeSlotSyncPayload
    │
    ├─ 同步村庄货币库存 CoinExchangeStatePayload
    │
    ▼
客户端收到后更新/重建槽位，并刷新货币交换面板状态
    │
    ▼
Screen 渲染
```

### 同步时机

| 场景 | 同步方式 | 网络包 |
|------|----------|--------|
| 打开 GUI | 全量同步槽位 + 村庄货币库存 | TradeSlotSyncPayload + CoinExchangeStatePayload |
| 添加到暂存区 | 增量同步 | PendingTradeSyncPayload |
| 移除暂存区物品 | 增量同步 | PendingTradeSyncPayload |
| 清空暂存区 | 增量同步 | PendingTradeSyncPayload |
| 常规交易成功 | 全量同步 + 增量同步 + 货币库存同步 | TradeSlotSyncPayload + PendingTradeSyncPayload + CoinExchangeStatePayload |
| 货币交换成功/失败 | 村庄货币库存同步 + GUI 反馈 | CoinExchangeStatePayload + TradeFeedbackPayload |
| 贸易周期刷新（清单可能重建） | 全量同步 + 货币库存同步 | TradeSlotSyncPayload + CoinExchangeStatePayload |

### 网络包设计

#### TradeSlotSyncPayload（服务端→客户端）

全量同步所有槽位：

| 字段 | 说明 |
|------|------|
| `containerId` | 菜单 ID |
| `sellSlotCount` | 出售槽位数量 |
| `buySlotCount` | 收购槽位数量 |
| `sellSlots` | 出售槽位数据列表（含 priceStacks、inputStacks、tradeType） |
| `buySlots` | 收购槽位数据列表（含 priceStacks、inputStacks、tradeType） |

#### PendingTradeSyncPayload（服务端→客户端）

增量同步暂存区：

| 字段 | 说明 |
|------|------|
| `containerId` | 菜单 ID |
| `pendingSlots` | 暂存槽位数据列表（包含每个槽位的 isBuy 标志） |
| `pendingCountMap` | 来源槽位 pendingCount 更新 |

#### TradeRequestPayload（客户端→服务端）

交易请求：

| 字段 | 说明 |
|------|------|
| `containerId` | 菜单 ID |
| `requestType` | 请求类型：ADD_BUY=0, ADD_SELL=1, REMOVE_ENTRY=2, CLEAR=3, REMOVE_BUY=4, REMOVE_SELL=5, CONFIRM=6 |
| `slotIndex` | 槽位索引（CONFIRM 时忽略） |

#### CoinExchangeRequestPayload（客户端→服务端）

货币交换请求：

| 字段 | 说明 |
|------|------|
| `containerId` | 菜单 ID |
| `exchangeType` | 兑换方向：铜→铁 / 铁→铜 / 铁→金 / 金→铁 |
| `exchangeAll` | 是否批量兑换最大次数 |

#### CoinExchangeStatePayload（服务端→客户端）

村庄货币库存同步：

| 字段 | 说明 |
|------|------|
| `containerId` | 菜单 ID |
| `copperCount` | 村庄当前铜板库存 |
| `ironCount` | 村庄当前铁币库存 |
| `goldCount` | 村庄当前金币库存 |

## 交易执行流程

### 确认交易

玩家点击确认按钮时：

```
Screen.onConfirmClick()
    │
    ▼ 发送 TradeRequestPayload(CONFIRM)
Menu.handleTradeRequest(CONFIRM)
    │
    ▼
Menu.executeTrade()
    │
    ├─ 获取 CommercialNodeData
    │
    ▼
直接从 pendingSlots 读取交易数据
    │
    ▼
TradeContractExecutor.executeMixedTrade(level, nodeData, player, pendingSlots, cycleIndex, blockPos)
    │
    ├─ 验证周期一致性
    ├─ 构建 TradePaymentPlan
    ├─ 验证玩家库存
    ├─ 验证村庄库存
    │
    ▼ 原子执行
TradeResult
    │
    ├─ 成功 → 清空暂存区 + 更新槽位库存 + 同步到客户端
    ├─ 跨周期 → 重建当期槽位清单 + 同步到客户端
    └─ 其他失败 → 发送 FailReason 翻译消息
```

### 价格计算

使用 `TradePricingService` 计算价格，组合基础价值、主题修正和市场因子：

```java
// TradeStationMenu.calculateSellPrice()
return TradePricingService.calculateFinalPrice(
    serverPlayer.serverLevel(),
    nodeData,
    stack,
    TradeSide.SELL_TO_PLAYER
);
```

服务端在交易执行时重新计算权威价格，确保客户端无法篡改。
