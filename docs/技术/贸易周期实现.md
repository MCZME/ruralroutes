## 设计概要

采用**惰性刷新**设计：
- 全局周期核心维护周期索引计算
- 玩家交互时检查节点是否需要刷新
- 避免主动轮询所有节点

## 核心类

### CycleManager (SavedData)

位置：`core/cycle/CycleManager.java`

```java
public class CycleManager extends SavedData {
    // 获取实例
    public static CycleManager get(ServerLevel level);
    
    // 计算周期索引
    public long getCycleIndex(long gameTime);
    
    // 判断是否需要刷新
    public boolean needsRefresh(long nodeTimestamp, long currentGameTime);
}
```

存储位置：`data/ruralroutes_cycle.dat`

第一阶段不存储数据，为第二阶段市场波动预留扩展点。

### Config 配置项

```java
// 周期长度（游戏日）
public static final ModConfigSpec.IntValue CYCLE_DAYS;  // 默认 3

// 刷新时刻枚举
public static final ModConfigSpec.EnumValue<RefreshTime> REFRESH_TIME;  // 默认 SUNRISE

public enum RefreshTime {
    SUNRISE(24000), NOON(6000), SUNSET(12000), MIDNIGHT(18000);
}
```

## 刷新逻辑

### 周期索引计算

```
周期索引 = gameTime / (cycleDays * 24000)
```

所有节点按同一全局周期边界刷新。

### 判断流程

```
玩家交互贸易站
    ↓
获取节点的 refreshTimestamp
    ↓
计算节点周期索引 vs 当前周期索引
    ↓
如果 当前 > 节点 → 刷新节点
    ↓
恢复库存到基准值（全量恢复）
更新 refreshTimestamp 为当前时间
```

### 触发点

`TradeStationBlock.useWithoutItem()` 中，数据校验通过后、打开菜单前调用：

```java
if (level instanceof ServerLevel serverLevel) {
    nodeData = CommercialNodeManager.checkAndRefreshCycle(serverLevel, pos, nodeData);
}
```

## 恢复规则

第一阶段实现 `full`（全量恢复）：
- 从主题模板重新初始化库存
- 出售物品：库存恢复到上限
- 收购物品：库存清零

第二阶段将实现 `category`（分类处理）和 `incremental`（增量恢复）。
