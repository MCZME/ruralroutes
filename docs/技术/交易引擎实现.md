# 交易引擎实现

## 核心思想

交易引擎负责执行贸易站交易，核心是把“GUI 上暂存的报价卡片”转换成服务端权威的物品转移计划。

核心原则：

- **服务端权威**：价格与库存都以服务端当前状态为准
- **不信任客户端**：客户端不发送价格明细，只发送操作请求
- **契约执行模式**：通过 `TradeContractExecutor` 统一执行单契约与混合交易
- **显式支付计划**：所有交易最终都落成 `TradePaymentPlan`
- **组件身份保留**：混合交易按 `TradeItemKey` 合并、校验和扣除，不再把带组件物品压成裸 `Item`

## 单一契约执行

适用场景：

- 货币交换面板
- 未来其他独立契约入口

流程：

```text
1. 按 ChunkPos 获取锁
2. 调用 contract.validateInput()
3. 调用 contract.execute()
4. 若成功，根据 TradeResult 更新村庄库存
5. 释放锁
```

库存更新规则：

- `consumed`：玩家消耗的物品，加入村庄库存
- `outputs`：玩家获得的物品，从村庄库存扣除

这里同样按 `TradeItemKey` 更新库存；若输出物品带组件而库存里只有无组件版本，会读取到无组件库存做容错更新。
其中货币条目在“加入村庄库存”时允许超过 `NodeStockEntry.max`，避免村庄货币库存已满时继续收款导致货币被虚空吞掉；普通物品仍保持上限钳制。

## 混合交易执行

适用场景：

- 贸易站主界面的买卖暂存区

流程：

```text
1. 验证周期一致性
2. 按 ChunkPos 获取锁
3. 遍历 pendingSlots，为每个槽位构建 TradePaymentPlan
4. 合并所有支付计划
5. 对合并后的物品列表做规范化合并
6. 校验玩家库存
7. 校验村庄库存
8. 原子执行玩家背包与村庄库存更新
9. 释放锁
```

### buildPlanForSlot

每个 `PendingTradeSlot` 会被翻译为明确的支付计划：

- `FIXED`：使用 `inputStacks`
- `CURRENCY_BASKET_DYNAMIC`：使用 `priceStacks`

这里不重新查主题配置，而是直接消费槽位同步过来的展示栈、价格栈和输入栈。

### consolidatePlan

`TradePaymentPlan.merge()` 只做列表拼接；真正的规范化发生在 `TradeContractExecutor.consolidatePlan()`：

- 相同 `TradeItemKey` 的条目会合并数量
- 带不同组件签名的同类物品不会被错误合并
- 合并后保留规范化的展示原型栈

这是本轮重构后新增的重要约束。

### 玩家库存校验

`validatePlayerInventory()` 现在不是简单按 `Item` 计数，而是做一遍“消耗模拟”：

- 先按“带组件需求优先”排序
- 在临时背包副本上逐条尝试扣除
- 扣不够的部分返回为 `shortfall`

这样可以避免两类问题：

- 普通物品和组件物品混合时重复计数
- 精确组件需求错误地消耗了不匹配的同类物品

### 玩家扣除

`removeItemFromPlayer()` 与校验共用同一套匹配规则：

- 需求带组件时，必须精确匹配 `TradeItemKey`
- 需求不带组件时，允许匹配任意同 `Item` 的栈

## TradePaymentPlan

```java
public record TradePaymentPlan(
    List<ItemStack> playerInputs,
    List<ItemStack> playerOutputs,
    List<ItemStack> villageInputs,
    List<ItemStack> villageOutputs
)
```

含义：

- `playerInputs`：玩家支付的物品
- `playerOutputs`：玩家获得的物品
- `villageInputs`：村庄支付的物品
- `villageOutputs`：村庄获得的物品

## TradeResult

```java
public record TradeResult(
    Reason reason,
    List<ItemStack> outputs,
    List<ItemStack> consumed,
    List<ItemStack> shortfall,
    int totalValueExchanged
)
```

`Reason`：

- `SUCCESS`
- `PLAYER_INSUFFICIENT`
- `VILLAGE_INSUFFICIENT`
- `INVALID_INPUT`
- `INVALID_REQUEST`
- `CYCLE_CHANGED`

## 与库存层的关系

交易引擎直接操作 `CommercialNodeData.stocks()`：

- 键：`TradeItemKey`
- 值：`NodeStockEntry`

混合交易执行后会原地更新 `CommercialNodeData` 的库存，再通过 `CommercialNodeManager.updateNodeData()` 持久化回区块。
普通物品的库存增加仍受 `NodeStockEntry.max` 限制；货币物品在入库时允许 `current > max`，用来保留超出本期收购容量的实际收款结果。

## 与定价层的关系

交易引擎本身不负责推导市场价格公式。

职责分工：

- `TradeStationMenu`：为 GUI 和默认货币篮准备当前展示价格
- `TradePricingService`：根据 `ItemStack + sourceKey` 计算权威价格
- `TradeContractExecutor`：只执行已经具象化的支付计划

这意味着：

- 固定交换不走市场价格
- 货币交换不走市场价格
- 只有动态货币篮类报价会在生成 `priceStacks` 时调用定价服务
