# 交易引擎实现

## 核心思想

交易引擎负责**执行贸易站交易**，使用服务端权威价格计算。

核心原则：
- **服务端权威**：价格由服务端在执行时重新计算
- **不信任客户端**：客户端不发送价格数据
- **直接读取暂存区**：服务端已有 pendingSlots 数据
- **支持混合交易**：允许同时购买和出售，计算净货币差额

```java
    ServerPlayer player,

    long currentCycleIndex,
    BlockPos blockPos

```

参数说明：
- `level`：服务端世界实例
- `nodeData`：商业节点数据
- `player`：服务端玩家实例
- `pendingSlots`：暂存区槽位列表（直接从 Menu 读取）
- `currentCycleIndex`：当前周期索引（用于跨周期检测）
- `blockPos`：贸易站位置

## 执行流程

```
1. 获取锁（基于 ChunkPos）
   - 使用 TradeLockManager 的 Striped Lock
   - 同一区块串行执行，不同区块并行

2. 验证周期一致性
   - 对比 currentCycleIndex 与实际周期索引
   - 跨周期 → 返回 CYCLE_CHANGED

3. 验证订单非空

4. 服务端权威价格计算
   - 遍历 pendingSlots
   - 对每个物品调用 TradePricingService.calculateFinalPrice()
   - 分别计算 sellValueTotal（购买物品总价）和 buyValueTotal（出售物品总价）

5. 计算净货币差额
   - netCoinValue = sellValueTotal - buyValueTotal
   - 正数：玩家需支付货币
   - 负数：玩家获得货币

6. 验证玩家物品库存
   - 检查玩家背包是否有足够的出售物品

7. 验证玩家货币库存
   - 仅当 netCoinValue > 0 时检查
   - 玩家是否有足够的货币支付净差额

8. 验证村庄库存
   - 只检查玩家购买的物品（村庄出售的）

9. 原子执行
   - 玩家背包 -= 出售物品，+= 购买物品
   - 根据净差额方向转移货币
   - 村庄库存更新
   - 通过 CommercialNodeManager.updateNodeData() 保存

10. 释放锁
```

## 价格计算

服务端在执行时重新计算每个物品的权威价格：

```java
for (PendingTradeSlot slot : pendingSlots) {
    TradeSide side = slot.isBuy() 
        ? TradeSide.SELL_TO_PLAYER 
        : TradeSide.BUY_FROM_PLAYER;
    
    int unitPrice = TradePricingService.calculateFinalPrice(
        level, nodeData, stack, side
    );
    int lineValue = unitPrice * slot.getBaseStock();
    
    if (slot.isBuy()) {
        sellValueTotal += lineValue;  // 玩家购买：支付货币
    } else {
        buyValueTotal += lineValue;   // 玩家出售：获得货币
    }
}
```

定价公式：
- `sellPrice = round(baseValue * sellModifier * marketFactor)`
- `buyPrice = round(baseValue * buyModifier * marketFactor)`
- `price = max(1, calculatedPrice)`

由于 `sellModifier > buyModifier`，混合交易通常会产生净货币差额。

## TradeResult

```java
public record TradeResult(
    Reason reason,
    List<ItemStack> shortfall,
    int totalValueExchanged  // 净货币差额
) {
    public enum Reason {
        SUCCESS,
        PLAYER_INSUFFICIENT,
        VILLAGE_INSUFFICIENT,
        INVALID_REQUEST,
        CYCLE_CHANGED
    }
}
```

注意：已移除 `VALUE_MISMATCH` 原因，因为混合交易允许净货币差额。

## 并发处理

同一区块的交易请求排队执行：

- 1024 个 ReentrantLock 复用（Striped Lock）
- 基于 ChunkPos 哈希分配锁
- 同一区块内串行执行，不同区块可并行
