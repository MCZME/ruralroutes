# 情报实现

将市场事件转换为玩家可读的传闻文本。

## 概述

当前情报系统不再直接把“目标 + 作用域 + 上涨/下跌”硬套进一组通用句式，而是改为：

1. 市场规则声明 `rumor_family`
2. 规则可选声明 `rumor_target_key`
3. 运行时 `MarketEvent` 继承这两项传闻元数据
4. `RumorEntry` 按 `family × scope` 选择模板

这样传闻文本从“价格播报”转为更接近商路消息，例如：

- “最近各地都在抢吃食。”
- “平原匠坊最近缺矿料。”
- “听说陶瓦窑新放出了些彩陶。”

## 目标

情报层负责表达：

- **说的是什么**：目标物品、物品池，或玩家视角下的别名
- **发生在哪里**：全局、群系、主题
- **是什么性质的消息**：缺货、热销、放货、出得多

它不直接显示：

- 具体价格数值
- `delta` 百分比
- 技术层规则 ID

## 数据结构

### `RumorEntry`

```java
public record RumorEntry(
    String targetNameKey,
    MarketScopeType scopeType,
    String scopeTargetName,
    RumorFamily family
)
```

字段说明：

| 字段 | 说明 |
|------|------|
| `targetNameKey` | 最终展示目标翻译键，可来自 `rumor_target_key`、物品翻译键或标签翻译键 |
| `scopeType` | `GLOBAL / BIOME / THEME` |
| `scopeTargetName` | 群系或主题的翻译键；全局为空 |
| `family` | 传闻语义族，决定文本模板组 |

### `RumorFamily`

当前实现四种语义族：

| Family | 语义 | 例子 |
|--------|------|------|
| `SHORTAGE` | 短缺、抢货、急缺 | “雪原旅站最近缺吃食。” |
| `SURPLUS` | 过剩、压仓、出得多 | “平原粮仓最近庄稼出得多。” |
| `DEMAND` | 走俏、热销、高价收 | “最近矿料在各地都走俏。” |
| `RELEASE` | 放货、到货、新出一批 | “听说陶瓦窑新放出了些彩陶。” |

## 市场事件到传闻的映射

### 规则层字段

`MarketEventRule` 新增两项可选字段：

| 字段 | 说明 |
|------|------|
| `rumor_family` | 这条规则属于哪类传闻 |
| `rumor_target_key` | 传闻文本使用的玩家视角目标名称 |

示例：

```json
{
  "target_ref": "#ruralroutes:pool/mineral",
  "rumor_family": "shortage",
  "rumor_target_key": "rumor.target.mineral"
}
```

### 运行时继承

`MarketStateGenerator` 在把规则展开为 `MarketEvent` 时，会把 `rumor_family` / `rumor_target_key` 一并复制过去。

因此客户端只要拿到 `MarketState`，就能独立生成完整传闻文本。

### 兼容回退

若规则未提供 `rumor_family`：

- `delta > 0` → 回退为 `DEMAND`
- `delta < 0` → 回退为 `SURPLUS`
- `delta == 0` → 回退为 `RELEASE`

若规则未提供 `rumor_target_key`：

- 精确物品 → 使用原版 `item.*` 翻译键
- 标签目标 → 回退到 `ruralroutes.tag.*` 标签翻译

**注意**：当前内置规则已经全部提供 `rumor_target_key`，因此系统池标签翻译只作为外部数据包兜底，不再是内置内容的主路径。

## 目标名称解析

`RumorGenerator.resolveTargetNameKey()` 的优先级：

1. `event.rumorTargetKey()`
2. 精确物品 → `item.{namespace}.{path}`
3. 标签 → `ruralroutes.tag.{namespace}.{path}`

### 玩家视角目标别名

当前内置规则常用以下别名：

| 翻译键 | 中文 |
|--------|------|
| `rumor.target.food` | 吃食 |
| `rumor.target.crop` | 庄稼 |
| `rumor.target.wood` | 木料 |
| `rumor.target.stone` | 石料 |
| `rumor.target.mineral` | 矿料 |
| `rumor.target.leather_fiber` | 皮货 |
| `rumor.target.ice_snow` | 冰货 |
| `rumor.target.nether_goods` | 下界货 |
| `rumor.target.ocean_goods` | 海货 |
| `rumor.target.end_goods` | 末地货 |
| `rumor.target.precious` | 稀罕货 |

### 宽池拆分

`#ruralroutes:pool/dye_decor` 这类系统池在传闻层不直接暴露统一叫法，而按具体规则拆成不同玩家视角目标：

| 场景 | `rumor_target_key` | 中文 |
|------|--------------------|------|
| 常见全局层 | `rumor.target.decor` | 染色商品 |
| 染料工坊 | `rumor.target.dye` | 染料 |
| 陶瓦相关 | `rumor.target.terracotta` | 彩陶 |

## 作用域名称解析

`scopeTargetName` 的来源：

| `scopeType` | 结果 |
|-------------|------|
| `GLOBAL` | 空 |
| `BIOME` | `biome.{namespace}.{path}` |
| `THEME` | `ruralroutes.theme.{path}` |

渲染时：

- 全局模板只接收目标参数
- 群系 / 主题模板接收“地点 + 目标”两个参数

## 模板体系

### 旧方案问题

旧实现使用一组通用模板，例如：

- `%s在%s价格%s`
- `%s的%s价格%s`

这会导致：

- `GLOBAL` / `THEME` / `BIOME` 句式不兼容
- 明明是“缺货”或“放货”，却只能表达成“上涨/下跌”
- 规则自身的语义被压平

### 当前方案

当前模板按 `RumorFamily × MarketScopeType` 分组。

例如 `SHORTAGE`：

| 作用域 | 示例模板 |
|--------|----------|
| `GLOBAL` | “最近各地都在抢%s。” |
| `BIOME` | “听说%s一带缺%s。” |
| `THEME` | “%s最近缺%s。” |

例如 `RELEASE`：

| 作用域 | 示例模板 |
|--------|----------|
| `GLOBAL` | “最近各地放出了一批%s。” |
| `BIOME` | “%s一带最近到了一批%s。” |
| `THEME` | “听说%s新放出了些%s。” |

### 模板选择

`RumorEntry.getDisplayText()` 会：

1. 根据 `family`
2. 再根据 `scopeType`
3. 从对应模板组中随机抽取一条

因此同一类消息会有轻微措辞变化，但不会跨语义乱跳。

## 闲谈

无市场事件时：

- `RumorGenerator.generateGossipRumor()` 返回闲谈 `RumorEntry`
- `RumorEntry.getDisplayText()` 直接翻译 `rumor.gossip.*`
- 不再通过传闻模板二次拼句

这样避免闲谈误走“市场句式”。

## 缓存

`RumorBoardBlockEntity` 会缓存 `RumorEntry.serialize()` 的字符串。

当前序列化格式：

```text
targetNameKey|scopeType|scopeTargetName|family
```

旧缓存若与当前结构不一致，会在下次打开传闻板时自然失效并重建，不影响市场状态本身。

## 当前边界

当前实现已确定：

- 内置规则应优先提供 `rumor_target_key`
- 传闻文本由 `rumor_family × scope` 决定，而不是通用价格句式

当前未处理：

- 按 `delta` 强弱细分“紧俏 / 疯涨 / 压仓”等语气层次
- 针对不同 `RumorFamily` 的颜色、图标或便签视觉差异
- 外部数据包的自动文案校验工具
