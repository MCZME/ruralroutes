# 村庄结构注入实现

实现贸易站建筑在村庄中的**必定生成**与**唯一生成**。

## 设计目标

| 目标 | 说明 |
|------|------|
| 必定生成 | 每个平原村庄至少有一个贸易站 |
| 唯一生成 | 每个村庄最多只有一个贸易站 |
| 兼容性 | 与其他修改村庄的模组（如 Waystones）共存 |

---

## 技术方案

### 核心原理

通过 Mixin 拦截 `JigsawPlacement.Placer.tryPlacingChildren` 方法，在首次访问 `houses` 模板池时替换为贸易站专用池。

### 村庄生成流程

```
town_centers (起始池)
    ↓ Jigsaw 方块连接
streets (街道)
    ↓ Jigsaw 方块连接
houses (房屋池) ← 我们在这里拦截
    ↓ 递归扩展
terminators (终止)
```

### 执行逻辑

```
村庄生成 →
  访问 houses →
    JigsawPlacementPlacerMixin.@ModifyArg 拦截 getHolder →
    替换为 trade_stations 池 →
    jigsaw 尝试放置 →
      ├─ 成功 → 递归 tryPlacingChildren(tradeStationPiece)
      │         → @Redirect(List.add) 检测到贸易站 piece 已真正加入 pieces
      │         → 设置 tradeStationPlaced = true
      └─ 失败 → tradeStationPlaced 保持 false，下次 houses 访问继续重试
  后续访问 houses →
    tradeStationPlaced = true →
    返回原始 houses 池 →
    正常房屋生成
```

> **两阶段检测**：不在池引用替换时设标志，而是等 jigsaw 真的把贸易站 piece 加入 `pieces` 后，通过 `@Redirect(List.add)` 记录放置成功。这样不会把“候选生成”误判成“实际落地”。

## 关键代码

### JigsawPlacementPlacerMixin.java

```java
@Mixin(targets = "net.minecraft.world.level.levelgen.structure.pools.JigsawPlacement$Placer")
public class JigsawPlacementPlacerMixin {

    @Unique
    private boolean ruralroutes$tradeStationPlaced = false;

    // 阶段二：检测贸易站 piece 是否真的成功加入 pieces
    @Redirect(method = "tryPlacingChildren", at = @At(value = "INVOKE", target = "Ljava/util/List;add(Ljava/lang/Object;)Z"))
    private boolean ruralroutes$markTradeStationPlaced(List<Object> pieces, Object addedPiece) {
        boolean added = pieces.add(addedPiece);
        if (!ruralroutes$tradeStationPlaced
            && addedPiece instanceof PoolElementStructurePiece piece
            && piece.getElement().toString().contains(RuralRoutes.MODID)
        ) {
            ruralroutes$tradeStationPlaced = true;
        }
        return added;
    }

    // 阶段一：替换 houses 池引用
    @ModifyArg(method = "tryPlacingChildren", at = @At(...))
    private ResourceKey<StructureTemplatePool> ruralroutes$forceTradeStationPool(
        ResourceKey<StructureTemplatePool> resourceKey
    ) {
        if (ruralroutes$tradeStationPlaced) {
            return resourceKey;
        }

        if (resourceKey.location().getPath().endsWith("/houses")) {
            ResourceKey<StructureTemplatePool> tradeStationPoolKey = ...;
            if (pools.getHolder(tradeStationPoolKey).isPresent()) {
                return tradeStationPoolKey;
            }
        }

        return resourceKey;
    }
}
```

## 兼容性分析

### 与 Waystones 模组共存

两个模组使用相同的技术方案，但互不冲突：

| 情况 | 结果 |
|------|------|
| Waystones 先执行 | 前几次 houses → Waystone（其内部计数），后续 houses → 贸易站（我们计数） |
| 我们先执行 | 前几次 houses → 贸易站（我们计数），后续 houses → Waystone（其内部计数） |
| 两者都计数 | 各自的建筑都在前几次 houses 连接中有机会生成 |

**原因**：两个模组都只在各自计数范围内替换 houses 访问，计数耗尽后返回原始池。因此各自建筑都会生成，只是顺序取决于 Mixin 优先级。

### 兼容性保证

- 使用 `@ModifyArg` 而非 `@Overwrite`，不覆盖整个方法
- 只在首次触发后标记，不影响后续生成
- 检查池是否存在才替换，避免破坏不支持的村庄类型
