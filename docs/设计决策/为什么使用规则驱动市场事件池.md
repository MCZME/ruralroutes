# 为什么使用规则驱动市场事件池

---
创建日期: 2026-05-15
最后更新: 2026-05-15
版本: v1.0
状态: 已确定
关联系统: 市场波动, 贸易周期, 物品池
---

## 决策

市场状态采用**规则驱动市场事件池**生成。

每个贸易周期刷新时，从数据包定义的 `market_event_rules` 中按权重抽取若干规则，展开为当前周期的 `MarketEvent` 列表，并组成 `MarketState`。

不采用“每周期固定 1 个全局波动 + 1 个区域波动”的写死槽位模型。

## 核心规则

### 规则即内容模板

每条规则至少定义：

- `target_ref`：受影响的物品或标签引用
- `scopes`：作用域展开规则
- `delta`：涨跌幅
- `weight`：被抽中的相对概率

规则本身不是运行时事件，而是“本周期可能发生什么”的模板。

### 周期内抽取，周期间重建

每次贸易周期刷新时：

1. 从规则池中按权重随机抽取若干规则
2. 将规则按作用域展开为具体事件
3. 生成当前周期唯一的 `MarketState`

上一周期的事件整体失效，不跨周期保留。

### 作用域由规则声明

规则支持三种作用域：

- `GLOBAL`：影响所有村庄
- `BIOME`：影响指定群系的村庄
- `THEME`：影响指定主题的村庄

作用域不再由系统预留固定槽位决定，而由内容规则自由组合。

### 价格影响可叠加

同一物品可同时命中多个市场事件。

各事件 `delta` 按加法叠加，再应用 `market.max_delta` 上限保护。

### 同周期结果稳定

市场状态使用周期索引作为确定性种子来源。

在同一套规则和主题数据不变的前提下，同一周期索引应生成同样的市场状态，保证服务端权威与客户端展示口径一致。

## 原因

1. **内容扩展更自然**：新增一个事件只需补一条规则，不必修改核心生成流程。
2. **作用域组合更灵活**：全局、群系、主题事件可以自由混用，不受固定槽位数量限制。
3. **更适合数据包驱动**：权重、目标、作用域都能通过 JSON 调整，便于后续平衡和模组整合。
4. **更符合当前实现**：现有 `MarketEventRuleCatalog`、`MarketStateGenerator`、`MarketStateResolver` 已围绕规则池展开。
5. **传闻板联动更直接**：传闻板按当前 `MarketState` 逐条生成情报，无需先把事件压缩回“全局/区域双栏”结构。

## 为什么不用固定双波动模型

固定双波动模型的优点是直观，但它有三个明显问题：

1. **扩展性差**：后续若想加入主题事件、节庆事件、稀有极端事件，就必须继续给系统增加新槽位。
2. **内容表达受限**：系统先规定“只能有几类事件”，内容只能往这个框里塞，反而限制设计空间。
3. **与现有代码分叉**：当前实现已经是规则抽取与作用域展开，如果继续保留固定双波动文档，只会造成设计与实现长期错位。

## 设计影响

### 对市场波动系统

“市场波动”从固定模板，转为“规则池抽取出的本周期事件集合”。

### 对配置结构

核心配置关注：

- 是否启用市场系统
- 每周期抽取多少条规则
- 叠加后的最大涨跌幅

单条事件的幅度与作用域，不再由公共配置统一生成，而由规则内容自身定义。

### 对后续内容设计

后续设计重点不再是“再补几个固定波动参数”，而是建立一组有层次的规则库，例如：

- 常见基础事件
- 群系特色事件
- 主题专属事件
- 低权重稀有事件
