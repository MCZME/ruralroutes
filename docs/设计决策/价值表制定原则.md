## 核心原则

以"获取难度"为基准，综合四个维度计算物品基础价值。

## 计算维度

### 一、获取难度

| 难度等级 | 说明 | 系数 | 示例 |
| :--- | :--- | :--- | :--- |
| 1 | 手可采，无限可再生 | 1-2 | 泥土、圆石、沙子、原木 |
| 2 | 需工具采集或简单加工 | 3-8 | 木板、木棍、燧石、羽毛 |
| 3 | 需熔炼或合成 | 10-25 | 铁锭、玻璃、面包、陶瓦 |
| 4 | 稀有矿或特定群系产物 | 30-60 | 钻石、金锭、黏液球、冰 |
| 5 | 下界/末地维度物品 | 60-120 | 烈焰棒、荧石、末影珍珠、潜影壳 |
| 6 | 极稀有/不可再生 | 120+ | 下界合金锭、附魔金苹果、鞘翅 |

### 二、可再生性

| 类型 | 说明 | 修正 |
| :--- | :--- | :--- |
| 无限可再生 | 可通过农场/刷怪塔量产 | ×0.8 |
| 有限可再生 | 需消耗不可再生资源来生产 | ×1.0 |
| 不可再生 | 开采后不恢复 | ×1.3 |

### 三、加工链路长度

| 加工层级 | 说明 | 附加值 |
| :--- | :--- | :--- |
| 原始 | 直接采集 | +0% |
| 初级加工 | 木板、木棍、熔炼产物 | +30% 基础值 |
| 中级加工 | 面包、箱子、铁轨 | +60% 基础值 |
| 高级加工 |蛋糕、附魔台、信标 | +100% 基础值 |

### 四、稀有度参数

物品在原版中的标准生成频率，作为微调因子（±20%）。

## 计算公式

```
物品基础价值 = [难度系数基准值 × 加工链路附加值] × 可再生性修正 × 稀有度微调
```

最终价值取整数，便于显示和计算。

## 举例试算

| 物品 | 难度系数 | 加工附加值 | 可再生性 | 稀有度微调 | 基础价值 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 泥土 | 1 | +0% | ×0.8 | ×1.0 | ≈ 1 |
| 圆石 | 1 | +0% | ×1.0 | ×1.0 | ≈ 1 |
| 原木 | 1.5 | +0% | ×0.8 | ×1.0 | ≈ 1-2 |
| 小麦 | 2 | +30% | ×0.8 | ×1.0 | ≈ 2-3 |
| 铁锭 | 3 | +60% | ×1.3 | ×1.0 | ≈ 6-8 |
| 钻石 | 4.5 | +0% | ×1.3 | ×1.1 | ≈ 6-7 |
| 末影珍珠 | 5 | +0% | ×1.0 | ×1.2 | ≈ 6-7 |
| 下界合金锭 | 6 | +100% | ×1.3 | ×1.5 | ≈ 23 |

注：以上数值为框架示例，具体系数可调整。

## 为什么不用原版村民交易价

原版交易存在大量"玩家福利"定价（如制箭师换木棍给绿宝石），缺乏内在逻辑。

本模组需要一套更有内在逻辑的价值表，让交换感更公平。

## 实现规范

基于以上原则，价值表采用 NeoForge DataMaps 实现。

### 实现形式

- **DataMaps**：使用 NeoForge 的 DataMap 机制，将价值数据附加到物品注册表。
- **标签原生支持**：数据文件中可直接使用标签 ID（如 `#ruralroutes:pool/stone`），自动匹配标签内所有物品。
- **热重载**：天然支持 `/reload` 命令重新加载数据。
- **数据包合并**：多个数据包的价值定义自动合并，后加载覆盖先加载。

### 文件格式

位置：`data/<namespace>/data_maps/item/value.json`

```json
{
  "values": {
    "#ruralroutes:pool/stone": { "value": 1 },
    "#ruralroutes:pool/crop": { "value": 5 },
    "minecraft:iron_ingot": { "value": 18 },
    "minecraft:diamond": { "value": 60 }
  }
}
```

#### 字段说明

| 字段 | 必需 | 类型 | 说明 |
| :--- | :--- | :--- | :--- |
| `values` | 是 | 对象 | 物品/标签到价值的映射。键为物品 ID 或标签 ID（以 `#` 开头）。 |
| `value` | 是 | 整数 | 物品的基础价值数值。 |

#### 键的格式

| 格式 | 示例 | 说明 |
| :--- | :--- | :--- |
| 物品 ID | `minecraft:iron_ingot` | 精确匹配单个物品 |
| 标签 ID | `#ruralroutes:pool/stone` | 匹配标签内所有物品 |

> **注意**：精确物品 ID 优先于标签。若某物品同时被标签和精确 ID 定义，使用精确 ID 的值。

### 默认价值

未匹配到任何价值规则的物品，使用**模组配置文件**中设定的默认价值：

```toml
# config/ruralroutes.toml
values.default = 1
```

- 该值对所有未明确定义的物品生效。
- 仅能通过修改配置文件调整，不受数据包文件影响。

### 覆盖逻辑

DataMaps 内置合并机制：
- 后加载的数据包值覆盖先加载的值
- 精确物品 ID 优先于标签
- 可通过 `remove` 字段排除特定物品

**整合包覆盖示例**：

```json
// data/mypack/data_maps/item/value.json
{
  "values": {
    "#ruralroutes:pool/stone": { "value": 10 }
  }
}
```

查询圆石时，使用覆盖后的值 `10`。

### 查询接口

```java
// 查询基础价值
int baseValue = ValueTableManager.queryBaseValue(itemStack);

// 计算静态价格（基础价值 × 主题修正系数）
int price = ValueTableManager.calculateStaticPrice(itemStack, 1.2f);
```

此方法仅在需要计算价格时调用（如打开贸易站界面、模拟交易结果等），不会每 Tick 执行。

### 配置示例

模组提供一套完整的默认价值规则，涵盖所有原版主要物品。整合包作者可在此基础上：

- **按物品池整体调整**：创建数据包，用标签 ID 修改整个池的基础价值。
- **精确调整**：用物品 ID 单独调整某个物品的价值。
- **与其他模组交互**：任何注册物品均可通过 ID 或标签引用，自动纳入价值体系。
