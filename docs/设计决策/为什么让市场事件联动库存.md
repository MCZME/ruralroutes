# 为什么让市场事件联动库存

---
创建日期: 2026-05-15
最后更新: 2026-05-17
版本: v1.0
状态: 已确定
关联系统: 市场波动, 村庄库存规则, 贸易周期, 交换机制
---

## 决策

市场事件不仅影响价格，也影响当期库存生成基线。

影响时机限定为：

- 节点首次创建
- 贸易周期刷新后重建当期库存

周期内玩家买卖只继续修改实时库存，不再反复重算市场库存修正。

## 核心规则

### 价格字段与库存字段分离

`MarketEventRule.delta` 继续只表示价格涨跌幅。

库存联动使用独立字段：

- `stock.sell`：影响出售给玩家的初始现货量
- `stock.buy`：影响村庄向玩家收货的容量

这样可以避免“高价必缺货、低价必放货”被写死成单一体验。

### 库存按双基线生成

库存初始化先得到两条基线：

- **出售基线**：该物品本期一开始能卖给玩家多少
- **收购基线**：该物品本期最多还能向玩家收多少

再合并为现有共享库存结构：

- 仅出售物品：`current = sellBase`，`max = sellBase`
- 仅收购物品：`current = 0`，`max = buyBase`
- 同时出现在买卖两侧：`current = sellBase`，`max = sellBase + buyBase`

### 主题模板先给基准，市场状态再做周期修正

`ThemeTemplate.stock` 仍负责声明库存基准范围。

市场事件命中后：

1. 先从主题模板随机出本期基础库存
2. 再按 `stock.sell` / `stock.buy` 分别修正出售基线和收购基线
3. 最后写入 `StockEntry(current, max)`

### 命中与叠加复用市场事件解析规则

库存联动与价格联动使用同一套匹配条件：

- 目标命中 `target_ref`
- 作用域命中 `GLOBAL / BIOME / THEME`

多条事件同时命中时：

- `stock.sell` 按加法叠加
- `stock.buy` 按加法叠加
- 两者分别应用 `market.max_stock_delta` 上限

### 传闻语义不直接绑定库存方向

`SHORTAGE / SURPLUS / DEMAND / RELEASE` 负责传闻表达。

库存方向由规则的 `stock` 字段显式声明，不从 `rumor_family` 隐式推导。

这让内容作者可以保留更灵活的组合，例如：

- 高价但仍有一定现货
- 放货但收购容量下降
- 热销主要体现在收购空间上，而不是现货骤减

## 原因

1. **修复表达落差**：传闻板说“缺货”“放货”时，玩家实际看到的库存也应有对应变化。
2. **保留内容灵活性**：价格与货量相关，但不必永远强绑定。
3. **适配现有实现结构**：库存只在节点创建和周期刷新时初始化，正适合做周期级一次性结算。
4. **数据包作者更易理解**：价格字段与库存字段分开后，规则语义更直观。

## 为什么不在周期内动态重算

如果把市场库存修正做成周期内实时作用，会带来三个问题：

1. 玩家交易后还要重新套市场因子，库存解释会变得绕。
2. `current` 与 `max` 的更新责任会和交易执行链混在一起。
3. 服务端校验、GUI 显示和展示柜物品选择都会变得更复杂。

因此本次设计把市场库存联动严格限定为“本期初始库存生成规则”，不参与周期内的每次库存变动。

## 设计影响

### 对规则结构

`MarketEventRule` 新增可选 `stock` 字段。

```json
{
  "delta": 0.15,
  "stock": {
    "sell": -0.35,
    "buy": 0.10
  }
}
```

### 对系统边界

- 市场状态模块负责解析库存修正
- 节点管理模块负责在创建/刷新时把修正应用到库存基线
- 交易执行模块不读取 `stock` 字段，不在周期内重算库存修正
