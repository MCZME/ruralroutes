# 测试补充计划

状态: 讨论
关联系统: 市场波动, 村庄库存规则, 贸易周期, 交换机制

## 背景

当前核心交易链路、市场事件系统和库存联动已经可以运行，但自动化测试覆盖仍然偏少。

最近又新增了“市场事件影响库存基线”的实现，如果没有补测试，后续继续调整规则结构、库存生成逻辑或周期刷新逻辑时，比较容易出现回归而不自知。

## 目标

补一批聚焦核心玩法规则的自动化测试，优先覆盖那些：

- 规则多、容易回归
- 看起来能跑，但细节容易错
- 一旦出错就会直接影响玩家体验或经济平衡

## 优先测试范围

### 市场事件解析

- 单条 `MarketEventRule` 的价格修正与库存修正能否正确解码
- `GLOBAL / BIOME / THEME` 三类作用域是否正确命中
- 多条事件叠加时，`delta` 与 `stock.sell / stock.buy` 是否分别按预期叠加
- `market.max_delta` 与 `market.max_stock_delta` 是否正确生效

### 库存初始化与周期刷新

- 节点首次创建时，是否会读取当前周期 `MarketState`
- 出售物品的初始库存是否正确应用 `stock.sell`
- 收购物品的收购空间是否正确应用 `stock.buy`
- 同一物品同时出现在买卖两侧时，`current / max` 是否按共享库存规则正确合并
- 周期刷新后是否重新抽样清单、重建库存，并应用新周期的市场修正

### 定价与库存边界

- 价格计算是否仍只读取 `delta`，不误读库存字段
- 库存变化是否不会反向影响报价
- 周期内玩家交易后，库存是否只按交易结果变化，不会再次套用市场库存修正

## 待定内容

- 测试主要写成单元测试、集成测试，还是两者都补
- 先补 `core/market` 与 `core/node` 的纯逻辑测试，还是直接做带世界状态的更高层测试
- 当前项目更适合使用哪套 NeoForge / JUnit 测试入口
- 是否要顺手为默认数据生成结果增加快照式校验

## 当前建议方案

在不改动原版链路的前提下，将测试框架拆成四层，分别覆盖“纯规则逻辑”“生成数据合法性”“带世界状态的行为验证”“人工体验验收”。

### 第一层：快速单元测试

- 使用 Gradle `test` 任务运行 JUnit 5
- 优先覆盖 `core/market`、`core/node`、`core/trade`、`core/theme` 中不依赖世界实例的纯逻辑
- 目标是让价格叠加、库存边界、共享库存规则、候选抽样、支付计划等逻辑能在几十秒内回归

### 第二层：生成数据校验

- 在 JUnit 中直接解析 `src/generated/resources` 下的主题、市场规则等 JSON
- 重点验证 Codec 是否可解码、字段是否齐全、权重与作用域是否满足基础约束
- 这层不是要替代快照，而是先保证“资源能读、结构没坏、基础约束没被破坏”

### 第三层：NeoForge GameTest

- 继续使用项目现有的 `runGameTestServer` 入口做世界状态验证
- 这一层适合放：
  - 周期刷新后是否重建库存
  - 节点首次创建时是否读取当前周期市场状态
  - 玩家交易后库存变化是否只受交易影响
  - 多人或服务端权威链路是否保持一致
- 这层应控制数量，只覆盖“纯逻辑无法证明”的关键行为

### 第四层：手工验收清单

- 用于补足自动化难以完全覆盖的体验项
- 重点包括：
  - 生存流程是否可持续推进
  - UI 与反馈是否清晰
  - 市场波动是否容易被玩家感知
  - 多人环境下的同步与重进服恢复

## 已落地的测试框架基线

当前建议先把自动化基线稳定下来，再逐步补高层测试：

- `build.gradle` 增加 JUnit 5 测试入口，统一使用 `./gradlew.bat test`
- 为加权规则抽样补独立可测试工具，避免核心规则选择逻辑被封在资源目录类内部
- 为节点共享库存规则补独立可测试模型，明确 `current = sellBase`、`max = sellBase + buyBase`
- 补生成数据合法性校验，优先验证主题与市场规则 JSON

## 推荐目录与职责

### `src/test/java`

- 放快速单元测试与数据校验测试
- 每个核心包对应一个测试包，尽量与主代码结构平行

### `src/test/resources`

- 预留给后续自定义测试样例 JSON、最小配置样本、异常输入样本

### `runGameTestServer`

- 放需要世界状态、区块数据、方块实体交互或服务端周期推进的验证

## 补测顺序建议

### 第一批：立即补

- 市场价格调整叠加、上限保护、方向与显示
- 市场库存调整叠加、独立买卖因子、边界取整
- 节点共享库存规则
- 主题物品引用与候选组基础约束
- 规则抽样逻辑
- 已生成主题与市场规则的可解码性

### 第二批：下一步补

- `CommercialNodeManager` 的库存初始化结果
- 周期刷新前后节点数据重建
- 特产注入与库存同步
- 价格计算中主题修正与市场修正的组合

### 第三批：GameTest 补齐

- 贸易站首次创建节点时读取当前周期状态
- 周期推进后节点重新抽样与库存恢复
- 玩家交易对库存和支付计划的服务端权威校验
- 刷新命令与手动触发刷新链路

## 覆盖矩阵补充

### 市场事件解析

- 纯逻辑层验证 `delta` 与 `stock.sell / stock.buy` 的独立叠加
- 数据层验证默认规则 JSON 可解码且作用域目标有效
- GameTest 层验证实际周期状态是否被节点与价格链路正确消费

### 库存初始化与周期刷新

- 纯逻辑层优先验证共享库存规则与库存边界
- 业务层继续补 `CommercialNodeManager` 初始化/刷新结果
- GameTest 层验证真实节点在世界中随周期变化的表现

### 定价与库存边界

- 单元测试验证库存修正不会直接污染价格修正对象
- 后续补 `TradePricingService` 组合测试，确认价格只读取 `delta`
- GameTest 验证周期内交易只消耗库存，不重复叠加市场库存修正

## 执行命令建议

- 快速回归：`./gradlew.bat test`
- 世界行为验证：`./gradlew.bat runGameTestServer`
- 数据变更后：先跑 `test`，再视影响范围补跑 `runData` 与 `runGameTestServer`

## 备注

- 这批测试优先服务于“市场事件 + 库存联动 + 周期刷新”三条链路的稳定性
- 不要求一口气补齐所有系统测试，可以先做最容易回归的核心用例
