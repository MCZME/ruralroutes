## 背景

本模组已经在主题模板、价值表、商业节点生成、市场事件规则中广泛使用标签引用，例如：

```text
#ruralroutes:pool/crop
#minecraft:axes
```

NeoForge 标签来自数据包，支持 reload。市场系统第二阶段会继续使用 `targetRef` 保存标签引用，因此需要明确标签查询、缓存和失效策略。

## NeoForge 相关约束

根据 NeoForge 1.21.1 标签文档：

- 标签通过数据包定义和加载。
- 标签内容可能在资源重载后变化。
- 频繁获取标签内容时，应缓存标签 entries。
- 标签缓存应在 `TagsUpdatedEvent` 发生时失效。

因此，任何缓存后的“标签 -> 物品集合”结果都不能跨 reload 假定稳定。

## 使用边界

### 可以直接查询的场景

单次物品命中检查可以直接使用：

```java
ItemStack.is(TagKey<Item>)
```

例如市场价格解析时判断某个 `ItemStack` 是否命中 `#ruralroutes:pool/crop`，不需要先展开整个标签。

### 需要缓存的场景

以下场景应使用统一缓存：

- 同一 tick 或同一 GUI 多次检查同一个标签。
- 需要从标签列出所有物品。
- 调试界面、传闻预览、配置工具需要展示标签内容。
- 市场规则校验需要批量检查标签是否为空。
- 后续主题、库存、市场系统出现重复的标签展开逻辑。

## 设计原则

- 市场事件规则加载时只校验 `targetRef` 字符串格式。
- 市场事件生成时不展开标签为具体物品。
- `MarketEvent.targetRef` 保留原始 `#namespace:path` 引用。
- 价格解析阶段再判断物品是否命中标签。
- 所有“标签 -> 物品集合”缓存必须集中管理。
- 缓存必须监听 `TagsUpdatedEvent` 并清空。
- 不允许多个系统各自维护无法统一失效的标签缓存。

## 建议工具类

建议新增统一工具：

```java
public final class TagLookupCache {
    public static boolean matchesItem(ItemStack stack, String itemRef);

    public static Set<Item> getItems(String tagRef);

    public static void invalidate();
}
```

职责：

- 解析 `#namespace:path` 与 `namespace:path` 引用。
- 单物品引用直接比较物品 ID。
- 标签引用使用 `TagKey<Item>` 进行命中检查。
- 需要展开标签内容时缓存不可变 `Set<Item>`。
- 在 `TagsUpdatedEvent` 时清空缓存。

## 事件监听

建议监听：

```java
TagsUpdatedEvent
```

处理：

```java
TagLookupCache.invalidate();
```

如果后续拆分出多个缓存，例如市场专用缓存、主题专用缓存，也应由统一入口统一失效。

## 对市场系统的影响

### MarketEventRuleCatalog

- 只校验 `target` / `targetRef` 格式。
- 不在规则加载时展开标签。
- 不因为标签当前为空就永久拒绝规则，除非明确设计为严格校验模式。

### MarketStateGenerator

- 只把规则展开为 `MarketEvent`。
- 不把标签目标拆成多个物品事件。
- 不缓存标签内容。

### MarketStateResolver

- 判断 `targetRef` 是否命中物品时，应通过统一工具。
- 返回的 `matchedEvents` 仍然保留原始 `targetRef`。

- 是否需要调试命令输出当前缓存的标签内容。
- 是否要顺手改造 `CommercialNodeManager` 中现有的标签展开逻辑，避免重复代码。
